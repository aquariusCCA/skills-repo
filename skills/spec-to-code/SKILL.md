# Spec To Code

使用這個 skill 將需求描述落地成可驗證的程式碼變更。目標不是只產生片段或建議，而是在足夠理解需求與專案脈絡後，完成最小可交付實作、補上合適驗證，並清楚回報結果與殘餘風險。

這是修改型 skill。除非使用者明確要求只規劃、只分析或先不要改檔，否則不要停在方案；建立足夠上下文後直接實作、驗證並回報。

## 建議輸入

能提供越多越好，但不必每次都完整提供。

- `spec`：需求描述、issue、ticket、PRD、TODO、bug 描述或驗收條件。
- `entry`：入口檔案、模組、路由、端點、元件、頁面、命令、測試、錯誤位置或相關關鍵字。
- `acceptance_criteria`：可觀察的完成條件，例如 UI 行為、API 回應、資料更新、錯誤處理、相容性或效能要求。
- `constraints`：限制修改範圍、不要碰的檔案、技術棧、相容性、風格、資料格式、時間或交付格式。
- `test_scope`：指定要新增或執行的測試、build、lint、typecheck、手動驗證或不需測試的理由。
- `output_mode`：可選。預設 `apply-in-place`；若使用者要先審核草案、不要直接修改原檔或指定輸出目錄，使用 `artifact-dir`。

## 核心規則

- 先把需求轉成工程上可驗收的條件。若需求已有明確驗收條件，直接使用；若沒有，從使用者文字與現有系統行為推導最小合理條件，並在回報中標明假設。
- 只在關鍵需求不明、不同解讀會導致高成本或不可逆變更時提問。能用保守假設前進時，先前進並標明假設。
- 先理解再改動。陌生專案、跨檔流程或入口不清楚時，使用 `source-code-reading` 的最小閉環方法找出相關流程。
- 實作要貼近既有架構、命名、測試風格與錯誤處理。不要為了單一需求引入新框架、大型抽象或無關重構。
- 控制變更範圍。不要把需求實作、全面格式化、架構重寫、命名整理與無關 bug 修正混在一起。
- 保護既有行為。除非 spec 明確要求變更，不要改公開 API、資料格式、排序、預設值、權限、交易邊界、錯誤語意或副作用。
- 對資料庫、外部 API、背景工作、排程、事件、feature flag、權限與遷移要特別保守；先確認現有模式與相容性。
- 修改前注意工作區可能已有使用者變更。不要還原或覆蓋無關變更；若相關檔案已有變更，讀懂後在其上工作。
- 驗證應和風險相稱。窄需求跑目標測試；跨模組或使用者可見功能至少跑相關測試、build、typecheck 或 lint；無法執行時要說明原因。

## 實作流程

1. 用一句話確認要交付的行為，以及最小驗收條件。
2. 整理已知輸入：`spec`、`entry`、`acceptance_criteria`、`constraints`、`test_scope`、`output_mode`。
3. 盤點專案入口：
   - 若有明確入口，先讀入口與直接相關依賴。
   - 若沒有入口，用 `rg --files`、manifest、路由、測試與關鍵字搜尋找保守起點。
   - 若流程陌生或跨檔，使用 `source-code-reading` 的最小閉環方法追到足以實作為止。
4. 將需求拆成小步驟：資料模型、API、服務邏輯、UI、狀態管理、驗證、錯誤處理、測試與文件。只保留本次需要的步驟。
5. 選擇最小可交付方案。若有多種方案，偏好符合既有模式、風險低、容易測試且改動少的方案。
6. 修改程式碼。每個改動都應能回到 spec 或驗收條件，不做無關整理。
7. 更新或新增測試。優先補能覆蓋新行為或 bug 修復的窄測試；若既有測試框架不足，使用最貼近專案慣例的驗證方式。
8. 執行目標驗證命令。若失敗，判斷是新實作問題、測試需要同步、既有失敗或環境問題，能修就修，不能修就清楚回報。
9. 回報變更內容、驗收條件如何被滿足、執行過的驗證、未驗證項目與殘餘風險。

## 需求不清時

使用以下順序處理模糊需求：

1. 從現有測試、UI、API schema、型別、文件與相似功能找本專案慣例。
2. 將模糊文字轉成保守假設，例如「預設不改公開 API」、「沿用現有錯誤格式」、「只影響指定頁面」。
3. 若假設錯誤會造成資料遷移、破壞相容性、安全風險、跨團隊契約改變或大量重工，先問使用者。
4. 若可安全前進，實作最小合理版本，並在回報中列出假設與可調整點。

不要把「需求不完整」當成停止工作的理由，除非缺口會阻擋正確實作。

## 測試與驗收

根據變更類型選擇驗證：

- 純邏輯或 bug 修復：新增或更新單元測試，並跑目標測試。
- API 行為：測 controller/handler/service，確認成功、錯誤、空值、權限與相容格式。
- UI 行為：測元件、互動、狀態轉換或端到端流程；若沒有測試框架，至少做 build/typecheck 並列出手動驗收步驟。
- 資料存取：測 mapper/repository/query 邏輯，確認交易邊界、排序、分頁、空資料與 migration 需求。
- 設定或整合：跑最接近的 build、lint、typecheck、整合測試或 smoke test。

若使用者明確要求不要加測試，仍應執行可行的最低成本驗證，並回報未覆蓋風險。

## 停止條件

滿足以下任一條件即可停止擴張範圍：

- 已完成 spec 指定的最小可交付行為，且相關驗證通過。
- 已完成可安全交付的部分，剩餘項目需要新的產品、資料、權限、架構或外部系統決策。
- 發現 spec 與現有公開契約、資料模型或安全要求衝突，且不能用保守假設解決。
- 關鍵資訊只能由未提供的執行時環境、資料庫內容、第三方服務或使用者決策確認。

## 預設忽略

除非與 spec 直接相關，否則不要優先修改：

- `node_modules`、`dist`、`build`、`.next`、`coverage`。
- lock file、編譯輸出與大型生成檔。
- 與本次需求無關的測試、文件、樣式與設定。
- 第三方函式庫原始碼。
- 只為風格一致而產生的大範圍格式化。

## 輸出指引

預設使用 `apply-in-place` 模式：直接修改專案中的目標檔案、測試與必要文件。控制台只輸出摘要、檔案清單、驗證結果與風險，不要貼出完整大型檔案內容。

若使用者要求先產出候選實作、不要直接修改原檔、要先審核草案，或明確指定輸出目錄，使用 `artifact-dir` 模式：

- 預設輸出到 `spec-to-code/<task-slug>/`，除非使用者指定其他目錄。
- 盡量保留原始相對路徑，讓檔案對應清楚。
- 補一份 `spec-to-code/<task-slug>/README.md`，說明需求、假設、檔案對應、如何套用、建議驗證命令與已知風險。
- 控制台只回報 artifact 位置、重點摘要、驗證狀態與需要使用者決策的點。

回報應精簡但包含工程上必要的資訊：

- 實作了哪些需求與核心變更。
- 改了哪些檔案。
- 驗收條件如何被滿足。
- 執行了哪些驗證命令與結果。
- 哪些驗證沒有執行，以及原因。
- 若有，列出假設、殘餘風險或需要使用者決策的點。
