# Output Contracts

這份 reference 提供完整輸出格式、失敗輸出契約，以及可直接套用的輸出骨架。只有在需要產生較正式、可交接的閱讀結果時再讀。

## 一、詳細輸出契約

除非使用者明確要求超短答，輸出至少要包含以下固定段落：

1. **這次要回答的問題**
2. **範圍與入口**
3. **專案邊界**
4. **實際閱讀 / 搜尋過的檔案**
5. **閱讀到的主路徑**
6. **已確認**
7. **合理推論**
8. **待驗證**
9. **證據衝突**（若有）
10. **目前停在哪裡**
11. **下一步**

### 每條核心結論的格式

- **[已確認]** 結論
  - 證據：`檔案路徑:行號` / `檔案路徑#符號名` / 執行時證據簡述
  - 說明：為什麼這能支持該結論

- **[合理推論]** 結論
  - 依據：哪些檔案 / 設定 / 呼叫關係支持這個推論
  - 缺口：還差哪個證據
  - 原因：為什麼目前只能推論

- **[待驗證]** 問題
  - 缺口：缺哪個證據
  - 影響：這個缺口會影響哪段結論
  - 下一步：應該讀哪個檔案、看哪個設定、檢查哪個執行時資訊

- **[證據衝突]** 問題
  - 衝突來源：哪些證據互相矛盾
  - 可能原因：profile、runtime、版本、環境、feature flag、generated code 等
  - 解衝突方式：最小需要補什麼證據

### 最低要求

- 重要結論不得無證據
- 搜尋命中不得直接當證據
- 若證據只到定義層，必須明說尚未確認生效
- 若缺乏行號，可用 `檔案路徑#方法名/類別名/段落名`
- 若證據來自日誌、錯誤訊息、執行輸出，必須標示為執行時證據
- 若存在主要替代路徑未排除，不得把結論寫成唯一事實

## 二、失敗輸出契約

若 2 到 3 輪入口追查後仍無法形成閉環，必須輸出失敗分類，而不是假裝看懂。

失敗分類可用以下之一：

- **入口未定位**
- **多候選未排除**
- **runtime 決定行為**
- **設定 / env 缺失**
- **generated code 缺失**
- **外部系統邊界**
- **檔案不足**
- **跨服務鏈未完整提供**
- **證據互相衝突**
- **當前問題定義過大**

失敗輸出至少要包含：

- 目前做到哪裡
- 卡在哪裡
- 失敗分類
- 影響範圍
- 最小下一步

```text
2. 範圍與入口
3. 主路徑
4. 已確認
5. 合理推論
6. 待驗證
7. 下一步
```
