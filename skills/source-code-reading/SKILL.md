# 原始碼閱讀

使用這個 skill 來建立有證據支撐的既有程式碼理解。目標是說明程式目前實際做了什麼、證據在哪裡、還有哪些不確定，以及下一步最值得閱讀什麼。

這個 skill 以先理解為主。如果使用者要求修復、重構、遷移或實作，先用這個 skill 建立相關上下文，再繼續完成使用者要求的工程工作。

## 何時使用

以下請求適合使用這個 skill：

- 閱讀或解釋某個檔案、類別、方法、元件、API、頁面、工作或模組。
- 追蹤某個功能、按鈕、路由、端點、批次工作、呼叫鏈或資料流。
- 在修改前理解陌生或舊有專案。
- 產出可重複使用的模組或功能筆記。
- 為聚焦任務找出下一步應該閱讀的檔案。
- 在除錯、重構、遷移或重寫前，先建立局部流程或模組邊界的可驗證理解。

如果使用者的主要需求是架構設計、技術選型、大範圍重寫規劃或完整實作，不要只用這個 skill 回答。這些情況下，只把它用於程式碼理解的部分。

## 建議輸入

能提供越多越好，但不必每次都完整提供。

- `goal`：這次閱讀是為了什麼，例如理解、除錯前置、重構前置、功能追蹤。
- `entry`：入口檔案、類別、方法、路由、端點、模組、UI 動作或錯誤位置。
- `mode`：`file`、`feature`、`module`；若未提供，依請求自動判定。
- `max_depth`：可選。限制最多展開幾層依賴或呼叫鏈；若未提供，依最小閉環原則決定展開範圍。
- `constraints`：可選。像是只看前端、只看後端、只看 SQL、限制輸出長度、限制時間。
- `expected_output`：可選。預設為 Markdown 格式筆記；也可指定直接回答、呼叫鏈、資料流、下一步閱讀清單，或要求輸出到特定 `.md` 檔案。若使用者沒有指定輸出檔名，但任務適合可重用或較長的閱讀筆記，主動選擇清楚的 `.md` 檔案路徑並在回覆中列出。
- `project_boundary`：可選。若專案是 monorepo、多服務或多工作區，指出目前只看哪個 app、service、package、workspace 或子系統。

## 核心規則

- 盡可能從具體入口開始：`entry`、檔案、類別、方法、功能名稱、路由、端點、模組、UI 動作或錯誤位置。
- 如果沒有提供入口，先做輕量盤點：用 `rg --files` 列出檔案，檢查明顯的 manifest、路由、設定檔、workspace 邊界與模組入口，再選擇保守的起點；只有在任務仍然模糊、且無法做出保守選擇時才提問。
- 如果找到多個合理入口，先列出 2 到 4 個候選入口與判斷依據，選擇最保守、最接近使用者問題、最容易形成最小閉環的入口開始；其餘列為備選路徑，不要一開始就全部展開。
- 只展開與目標直接相關的依賴、呼叫方與被呼叫方。若使用者提供 `max_depth` 就依其限制；否則不要固定只追一層，而是追到足以解釋問題的最小閉環為止。
- 「最小閉環」是指：已能用一條可解釋的局部流程回答使用者問題，而不是停在半路。常見例子：
  - 前端互動：路由 / 事件 -> 元件 / store -> API -> 狀態更新 / 渲染。
  - 後端請求：controller -> service -> repository / mapper -> SQL / 外部 API -> response。
  - 批次工作：trigger -> job -> service -> persistence / side effect。
- 若專案是 monorepo、多服務或多工作區結構，先確認這次請求所屬的 app、service、package、workspace 或子系統邊界。除非問題本身就是跨服務流程，否則不要跨 workspace、跨 service 或跨 app 擴張閱讀範圍。
- 若問題本身涉及跨前後端、跨服務或跨工作區流程，先明確標出跨界點，再分段追蹤每一段最小閉環，不要直接把整個系統攤平。
- 區分證據狀態：已確認、合理推論、待驗證。
- 不要宣稱已閱讀沒有實際開啟或搜尋過的檔案。
- 不要從單一檔案、單一 controller、單一 component 或單一命名慣例推斷完整系統設計。
- 重要結論優先附上檔案與行號參考。
- 預設以 Markdown 格式輸出閱讀筆記；輸出長度應符合使用者請求。窄問題用精簡 Markdown 筆記，明確要求廣泛分析時再使用完整結構化筆記。
- 窄問題、短答案或一次性判斷可以直接在對話中輸出 Markdown；較長、結構化、可重用或跨檔案的閱讀筆記，預設寫入 `.md` 檔案，並在回覆中明確列出檔案路徑。
- 若使用者明確指定輸出路徑或要求建立檔案，照指定路徑寫入。若沒有指定，輸出路徑優先順序如下：
  1. 專案既有的 `docs/code-reading/`
  2. 專案既有的 `notes/code-reading/`
  3. 專案既有的 `docs/` 或 `notes/` 下合理子路徑
  4. 若都沒有，使用 `docs/code-reading/`
- 除非使用者明確要求超短答，輸出至少應包含以下最低欄位：
  - 範圍與入口
  - 實際閱讀或搜尋過的檔案
  - 已確認行為
  - 待驗證項目
  - 下一步閱讀或下一步行動
- 若目前已有足夠資訊可開始最小閉環閱讀，優先直接開始；不要為了補齊非必要欄位而中斷任務提問。
- 若必須提問，問題應最小化，只問會直接影響入口選擇、專案邊界或輸出形式的關鍵資訊。

## 停止條件

滿足以下任一條件即可停止擴張閱讀範圍：

- 已能直接回答使用者問題。
- 已形成足以解釋問題的最小閉環。
- 新展開的檔案只補充背景，沒有實質改變目前結論。
- 剩餘不確定性已明確落在執行時資訊、資料庫內容、環境設定、外部系統、生成碼或未提供的檔案上。
- 已碰到清楚的專案邊界；若再擴張只會進入不相關模組、其他服務或其他工作區。

## 預設忽略

除非與目前問題直接相關，否則不要優先展開：

- `node_modules`、`dist`、`build`、`.next`、`coverage`。
- lock file、編譯輸出與其他大型生成檔。
- 與目前入口無直接關聯的測試檔。
- 只做 re-export 的 barrel files。
- 純樣式檔，除非問題與 UI 呈現、class 切換或樣式條件有關。
- 與當前流程無直接關聯的第三方函式庫原始碼。
- 大型自動生成檔、schema 產物或 codegen 輸出，除非問題本身與生成流程或生成結果有關。

## 預設模式

根據使用者請求選擇模式。若使用者指定模式，就照指定模式執行。

- `file`：理解單一檔案的角色、結構、依賴、關鍵邏輯、風險與下一步閱讀方向。
- `feature`：追蹤使用者可見功能、端點、按鈕、路由、批次工作或跨檔流程。
- `module`：以一個有明確邊界的目錄、package、domain、功能群或子系統為閱讀單位，根據實際讀過的檔案整理模組責任、主要檔案、典型流程、依賴、耦合與缺口。

如果使用者只提供檔案路徑，預設使用 `file`。
如果使用者提供功能名稱加入口，預設使用 `feature`。
如果使用者提供目錄、package、workspace、service 或模組並要求總覽，預設使用 `module`。

### `module` 模式的意思

`module` 不是要求讀完整個專案，也不是把某個目錄底下每個檔案逐一摘要。它適合用在使用者想理解「一組相關檔案共同負責什麼」時：先從目錄、package、公開入口或代表性流程做保守掃描，再閱讀足以說明模組責任與典型流程的檔案。

適合使用 `module` 的情況：

- 使用者提供模組目錄，想知道它的責任與主要檔案，例如 `src/modules/auth`。
- 使用者提供後端 package，想理解 controller、service、repository / mapper、DTO/entity 的責任分布，例如 `src/main/java/com/example/order`。
- 使用者準備重構某個功能群，想先知道目前耦合、風險與下一步閱讀點，例如 `src/features/pdf-viewer`。
- 使用者想建立可重用筆記，而不是只回答某個按鈕、某個 API 或某個函式怎麼跑。
- 在 monorepo 中，使用者想理解某個 workspace、app 或 service 的局部責任，而不是整個倉庫。

不適合使用 `module` 的情況：

- 只有單一檔案要解釋：使用 `file`。
- 只追一個端點、按鈕、路由、錯誤流程或批次觸發鏈：使用 `feature`。
- 問題是完整系統架構、技術選型或大範圍重寫：不要只用這個 skill 回答；先把範圍縮到可讀的模組或流程。
- 使用者要求「把整個目錄所有檔案都摘要」：先確認這是清單盤點需求，否則仍以代表性檔案與最小閉環為主。

`module` 範例：

- `entry: src/modules/auth`：整理登入、登出、session 狀態、API client、route guard 或 store 的責任分布與典型流程。
- `entry: src/main/java/com/example/order`：整理 order package 中 controller、service、mapper / repository、DTO/entity 如何分工，並列出常見 request 到 persistence 的流程。
- `entry: src/features/pdf-viewer`：整理 PDF viewer 相關元件、composable / hook、store、rendering 或 worker 邏輯的邊界與耦合風險。
- `entry: jobs/invoice`：整理 invoice batch 的 scheduler / job / service / persistence / side effect 邊界與資料更新點。
- `entry: apps/web/src/features/profile`：整理 monorepo 中 web app 的 profile 功能責任、依賴與代表性流程。
- `entry: services/payment`：整理 payment service 的對外入口、主要流程與與其他服務的交界點。
- `entry: src/shared/form`：整理 shared form 模組提供哪些可重用元件或工具、依賴哪些驗證邏輯，以及哪些呼叫方最能代表它的使用方式。

## 入口與邊界判定

在正式展開閱讀前，先做保守判定：

- **單一明確入口**：直接從該入口開始。
- **沒有明確入口**：先做輕量掃描，從最可能形成最小閉環的入口開始。
- **有多個候選入口**：先列出候選入口與判斷依據，選最保守的起點；其餘保留為備選，不同步擴張。
- **monorepo / 多服務 / 多工作區**：先確認這次請求屬於哪個 app、workspace、package 或 service，再開始閱讀。
- **跨界流程**：先標出每個跨界點，例如前端 -> API、gateway -> service、service -> queue、service -> DB，再分段追蹤。
- **仍然模糊且無法保守決策**：才提問，而且只問會改變入口、邊界或輸出形式的最小問題。

## 閱讀流程

1. 用一句話確認目標：這次要理解什麼，以及為了什麼目的。
2. 整理已知輸入：`goal`、`entry`、`mode`、`max_depth`、`constraints`、`expected_output`、`project_boundary`。
3. 確認專案邊界與入口；若有多個候選入口，先列候選與判斷依據，再選擇起點。
4. 只在需要時做輕量專案掃描：manifest、workspace 設定、路由表、模組目錄、框架設定或符號搜尋。
5. 先閱讀入口檔案。
6. 只追蹤相關的呼叫方、被呼叫方、import、路由、API 呼叫、SQL / mapper、DTO / entity 定義、設定或 UI 事件，並追到形成最小閉環或達到 `max_depth` 為止。
7. 閱讀時追蹤證據狀態：
   - 已確認：可從已開啟檔案或命令輸出直接看見。
   - 合理推論：根據命名、框架慣例、目錄結構或部分呼叫關係推論。
   - 待驗證：重要結論受未讀檔案、缺少執行時脈絡、隱藏 SQL、生成碼、環境設定、多種可能流程或跨服務邊界阻擋。
8. 依使用者要求的格式回答；若未指定格式，預設輸出 Markdown 格式筆記。若是較長、結構化、可重用或跨檔案的筆記，寫入 `.md` 檔案並在回覆中列出路徑；若任務較廣，使用 `references/output-templates.md` 中對應的模板，但模板可依內容調整、合併或省略局部欄位，避免破壞閱讀流暢度。
9. 除非回答本來就需要非常簡短，結尾應指出最高價值的下一個閱讀目標或下一步行動。

## 技術線索

優先依照專案實際檔案與慣例判斷。以下只是提醒，不是固定技術棧清單。

- Java web / Spring / Spring Boot：檢查 controller-service-repository 或 mapper 流程、DI、DTO/entity 流動、驗證、交易邊界、例外處理與設定。
- Hibernate / MyBatis / SQL-heavy 專案：檢查 DAO/mapper XML 或註解、HQL/SQL、stored procedure、result mapping、交易邊界與 entity/DTO mapping。
- Vue / React / 前端專案：檢查路由、元件邊界、props/events、store/composables/hooks、API client、狀態轉換、表單處理與渲染條件。
- jQuery / 舊式前端：檢查 DOM 綁定、初始化流程、全域狀態、AJAX 呼叫、工具函式耦合、動態 HTML 生成與表單參數組裝。
- Batch / Scheduler / Worker：檢查 trigger、job、queue consumer、service、副作用、重試邏輯、持久化與外部整合點。
- Monorepo / 多服務：檢查 workspace 設定、app/service 邊界、shared package、跨服務 API / queue / schema 契約與部署切面。
- 未知技術棧：先檢查 manifest、workspace 設定與入口/設定檔，再套用假設。

## 輸出指引

預設輸出 Markdown 格式筆記，並為關鍵點附上證據。短答案與窄問題可以直接在對話中回答；較長、結構化、可重用或跨檔案的閱讀筆記，預設寫入 `.md` 檔案，並在回覆中明確列出檔案路徑，方便使用者用編輯器閱讀。

窄問題可以用精簡 Markdown 筆記直接回答，例如只包含「結論」、「證據」與「下一步」。任務較廣、要求可重用筆記或沒有指定輸出格式時，使用結構化 Markdown 筆記。

除非使用者明確要求超短答，輸出至少應包含：

- 範圍與入口。
- 實際閱讀或搜尋過的檔案。
- 已確認行為。
- 待驗證項目。
- 下一步閱讀或下一步行動。

結構化筆記應包含：

- 範圍與入口。
- 實際閱讀或搜尋過的檔案。
- 已確認行為。
- 合理推論，並清楚標示。
- 待驗證項目。
- 與請求相關的依賴、呼叫方、被呼叫方或資料流。
- 相關時才列出風險或可疑區域。
- 下一步檔案或行動，以及原因。

只有在目前請求適合完整筆記格式時，才讀取 `output-templates.md` 的詳細模板。只有在目前請求適合完整筆記格式時，才讀取 `references/output-templates.md` 的詳細模板。模板是起點，不是硬性格式；若某些欄位不適用、資訊量太少，或會讓筆記變得割裂，應省略或合併。
