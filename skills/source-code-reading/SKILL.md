# 原始碼閱讀（最終穩定版）

使用這個 skill 來建立**可驗證、可追溯、可交接**的既有程式碼理解。  
核心目標不是「快速說明這段程式大概在做什麼」，而是回答：

- 程式目前**實際**做了什麼。
- 哪些結論是**已確認**，證據在哪裡。
- 哪些只是**合理推論**，推論依據是什麼。
- 哪些仍是**待驗證**，缺口在哪裡。
- 下一步最值得閱讀、搜尋或最小驗證的是什麼。

---

## 一、何時使用

以下情況適合使用這個 skill：

- 閱讀或解釋某個檔案、類別、方法、元件、端點、頁面、job、模組。
- 追蹤某個功能、按鈕、路由、API、批次工作、資料流或呼叫鏈。
- 在修改、除錯、重構、遷移、重寫前，先理解既有實作。
- 產出可重複使用的功能筆記、模組筆記、閱讀報告。
- 判斷下一步應該讀哪個檔案、哪個符號、哪一層邊界。

以下情況**不要只靠這個 skill**直接回答：

- 架構設計、技術選型、完整重寫方案才是主要需求。
- 使用者已明確要求直接修復、改寫、遷移、實作。
- 任務核心不是理解既有程式，而是產出新設計或新程式。

這些情況下，本 skill 只負責「先理解既有程式」那一段。

---

## 二、建議輸入

能提供越多越好，但不必每次都完整提供：

- `goal`：閱讀目的，例如理解、除錯前置、重構前置、功能追蹤。
- `entry`：入口檔案、類別、方法、路由、端點、UI 動作、錯誤位置。
- `mode`：`file`、`feature`、`module`；若未提供，自動判定。
- `max_depth`：可選。最多展開幾層呼叫鏈或依賴。
- `constraints`：可選。例如只看前端、只看後端、只看 SQL、限制輸出長度。
- `expected_output`：可選。例如直接回答、結構化筆記、呼叫鏈、資料流、下一步清單。
- `project_boundary`：可選。若為 monorepo、多服務或多工作區，指出這次只看哪個 app、service、package、workspace。

---

## 三、規則優先級

若不同規則彼此衝突，依以下順序處理：

1. 使用者當前明確要求
2. 安全、唯讀、低副作用邊界
3. 本 skill 的明文規則
4. 專案既有慣例與目錄習慣
5. 一般框架慣例與命名推測

補充：

- 使用者要求「直接回答，不要建檔」時，優先直接回答。
- 使用者要求與安全邊界衝突時，先完成閱讀結論，不在閱讀階段直接越界執行。
- 專案慣例可參考，但**不得凌駕於證據**。

---

## 四、不可違反的核心原則

### 1. 先找入口，再追最小閉環
從最接近問題的入口開始：錯誤位置、路由、端點、UI 動作、job 觸發點、檔案或方法。  
沒有明確入口時，先做輕量盤點，再選最保守起點。  
目標不是讀最多檔案，而是形成**最小閉環**。

### 2. 最小閉環優先於大範圍掃描
最小閉環是指：已能用一條局部、可解釋、可追溯的流程回答使用者問題。

常見閉環：

- 前端互動：事件 / 路由 → 元件 / store / composable → API → 狀態更新 / 畫面渲染
- 後端請求：controller / handler → service → repository / mapper → SQL / 外部 API → response
- 批次任務：trigger → job / worker → service → persistence / side effect
- 跨服務流程：入口服務 → 契約邊界 → 下游服務 / queue consumer

### 3. 必須區分三層位置
閱讀時必須區分：

- **定義位置**：在哪裡宣告、定義、描述
- **註冊位置**：在哪裡被掛進系統或綁定
- **實際生效位置**：在當前問題中，真正被執行、匹配、注入、命中的位置

只看到定義，不等於已確認生效。

### 4. 證據分級是硬規則
所有結論只能落在三種狀態之一：

- **已確認**：有直接證據支撐
- **合理推論**：有依據，但仍未完全證實
- **待驗證**：目前缺證據或存在多種可能

搜尋命中、命名直覺、框架慣例、個人經驗，都**不能直接當已確認**。

### 5. 先靜態閱讀，必要時才做最小驗證
預設先用唯讀方式建立理解：

- 列檔
- 搜尋符號
- 讀檔
- 看設定
- 看測試
- 看錯誤訊息
- 看既有日誌

只有在**不做最小驗證就無法區分兩條以上關鍵路徑**時，才做唯讀、低風險、可回溯的最小驗證。

### 6. 優先縮邊界，不優先擴張閱讀量
monorepo、多服務、多工作區時，先確認這次請求屬於哪個 app、service、workspace、package。  
除非問題本身就是跨界流程，否則不要一開始就跨 service / workspace / app 擴張。

### 7. 多候選入口必須先排除，再展開
若找到多個入口、同名符號、多個實作、多條可能路徑：

- 先列候選
- 說明各自理由
- 選最保守、最接近問題、最容易形成閉環的起點
- 其餘保留為備選

不得因為第一個搜尋命中看起來合理，就直接當真。

### 8. 閱讀階段不滑入工程執行
本 skill 的主要目的，是建立理解，不是直接修改原始碼、修 bug、重構、部署、安裝、更新資料。  
若使用者真正需求已轉成工程執行，先完成閱讀結論，再切換到後續階段。

## 五、標準執行流程（硬流程）

每次閱讀至少依以下流程進行；除非使用者要求極短答，否則不要跳步。

### Step 0：確認目標
用一句話確認這次閱讀要回答什麼，以及是為了什麼。

### Step 1：確認邊界
先判定：

- 單一檔案
- 單一功能流程
- 單一模組 / package / 目錄
- 單一 service / workspace / app
- 或跨界流程

若為 monorepo / 多服務 / 多工作區，先縮到本次請求的局部邊界。

### Step 2：做最小盤點
只做有助於找入口的輕量盤點，例如：

- manifest / workspace 設定
- 路由表 / controller / API 定義
- 模組入口
- 框架設定檔
- 符號搜尋

不要一開始掃整庫。

### Step 3：選入口
入口優先順序如下：

1. 使用者提供的錯誤位置、堆疊、異常訊息、UI 操作、API 名稱
2. 路由、controller、handler、event binding、UI action 綁定位置
3. service、hook、composable、use case
4. repository、mapper、DAO、API client、query builder
5. DTO、entity、schema、契約、設定檔
6. 測試、fixture、日誌、文件

若有多個候選入口，先列候選與理由，再選最保守起點。

### Step 4：讀入口，不要只看名稱
閱讀入口檔案時，先回答：

- 這裡是不是實際入口，還是只有定義 / 轉發 / re-export？
- 它的輸入是什麼？
- 它把控制權交給誰？
- 它是否只是包裝層，而真正邏輯在下一層？

### Step 5：沿關鍵路徑追到最小閉環
只追與目前問題直接相關的：

- 呼叫方
- 被呼叫方
- import
- store / state
- API client
- mapper / SQL
- DTO / entity
- route / event
- config / feature flag
- 測試 / 樣例

追到足以回答問題的最小閉環即可。

### Step 6：對每段關鍵結論做證據分級
每條重要結論都必須明確標示：

- 已確認
- 合理推論
- 待驗證

並補上證據或缺口。

### Step 7：檢查是否該停止
出現以下任一情況即可停止擴張：

- 已能直接回答使用者問題
- 已形成最小閉環
- 新展開檔案不再改變結論
- 剩餘不確定性落在 runtime、資料內容、外部系統、環境或未提供檔案上
- 再展開只會跨到不相關模組或其他 service / workspace

### Step 8：輸出結果
輸出時至少要交代：

- 範圍與入口
- 實際讀過 / 搜過哪些檔案
- 已確認
- 合理推論
- 待驗證
- 下一步

## 六、必做閱讀核對清單

除非使用者明確要求超短答，否則每次閱讀至少應回答以下問題；若暫時答不出，也要交代缺口。

### 通用核對

- 入口是什麼
- 輸入是什麼
- 輸出是什麼
- 狀態在哪裡改變
- 副作用在哪裡發生
- 錯誤路徑是什麼
- 同步 / 非同步邊界在哪裡
- 設定是否影響行為
- 目前確認到的是定義、註冊，還是實際生效

### 後端流程額外核對

- request 先經過哪些 controller / filter / interceptor / middleware
- 核心業務邏輯在哪個 service
- persistence 實際落在哪個 repository / mapper / SQL
- transaction、權限、例外處理是否介入
- 是否有 cache、retry、排程、非同步或外部系統互動

### 前端流程額外核對

- 事件如何綁定
- 狀態由誰持有與更新
- API 從哪裡發出
- 回應如何映射回 UI
- route guard、dynamic import、lazy loading 是否改變流程
- 呈現差異是否由 class、條件渲染、動畫狀態影響

### SQL / 資料存取額外核對

- SQL 定義在哪裡
- SQL 由誰組裝或呼叫
- 參數如何綁定
- 結果如何映射
- 是否存在動態 SQL、stored procedure、view、trigger、resultMap
- 目前確認的是靜態 SQL，還是實際命中的 SQL

---

## 七、符號級最小閱讀模板

### 方法 / 函式
至少確認：

- 誰呼叫它
- 它呼叫誰
- 參數從哪裡來
- 回傳值去哪裡
- 是否存在 early return / 例外攔截 / 隱藏副作用
- 是否讀寫共享狀態、cache、DB、檔案、外部服務

### 元件 / 頁面
至少確認：

- props / route / store / composable 輸入
- 使用者操作如何觸發事件
- 狀態在哪裡改變
- 什麼條件影響渲染
- API / 子元件 / 外部 hooks 的邊界在哪裡

### 端點 / controller / handler
至少確認：

- 路由如何匹配到它
- request 驗證、授權、轉換在哪一層完成
- 核心邏輯交給哪個 service
- response 由誰組裝
- 失敗路徑如何處理

### repository / mapper / DAO
至少確認：

- 查詢真正來源是 XML、註解、query builder 還是 generated code
- 參數如何綁定到 SQL
- 結果如何映射到 entity / DTO
- 是否存在動態 SQL、條件拼接、resultMap、join 擴充

### 模組 / package / 目錄
至少確認：

- 模組責任是什麼
- 對外入口有哪些
- 典型流程如何流經主要檔案
- 哪些檔案是核心、哪些只是支援
- 邊界、耦合點、跨界點在哪裡

---

## 八、硬性反誤讀規則

以下行為預設禁止，除非已有直接證據：

- 只憑檔名、目錄名、類別名、方法名判定真實責任
- 只看到 interface / abstract class 就宣稱某實作一定被執行
- 只看到 DTO / entity / 欄位名就推定完整業務語意
- 只看到按鈕文字、template 標記、路由名稱就宣稱已追到事件鏈
- 只看到 mapper 介面或 SQL 片段就宣稱已確認實際命中的 SQL
- 只看單一測試就把它當全部行為證據
- 只看 barrel / re-export / index file 就宣稱已理解核心邏輯
- 把搜尋命中當成已確認
- 把框架慣例當成專案事實
- 把個人經驗、過往專案印象包裝成目前專案結論

另外，遇到以下情況時，必須額外保守：

### 1. 同名符號
若有多個同名 class / method / component / service / mapper：

- 不得只看第一個搜尋命中
- 必須交代為什麼排除其他候選
- 若尚未排除，結論只能列為待驗證或合理推論

### 2. 多實作 / 插件機制 / DI
若看到 interface、plugin contract、abstract class、strategy registry、bean 注入：

- 先列可能實作
- 再找註冊點、注入點、匹配條件
- 沒有注入 / 匹配證據，不得宣稱哪個實作一定生效

### 3. 動態映射 / 動態 SQL / 動態路由
若流程依賴 runtime config、profile、annotation、AOP、反射、dynamic import、feature flag、MyBatis 動態 SQL、ORM 映射：

- 先說明靜態可見部分
- 再說明哪段仍受 runtime 影響
- 必要時才做最小驗證
- 無法驗證時，必須明確降級為待驗證

## 九、動態行為的最小驗證策略

以下情況常無法只靠靜態閱讀完全確認；應優先做**最小、唯讀、低副作用**驗證，而不是直接假設。

### 1. Spring / DI / Annotation / AOP
優先檢查：

- annotation / interface / aspect 定義
- bean 註冊或條件裝配位置
- 實際注入點
- 攔截 / 匹配條件
- 本流程中的實際被呼叫點

### 2. MyBatis / ORM / SQL-heavy
優先檢查：

- mapper / repository 介面
- XML / 註解 / query builder / generated SQL
- service 如何組參數
- resultMap / entity / DTO 映射
- 是否有動態 SQL、stored procedure、view、trigger

### 3. 前端動態流程
優先檢查：

- route 定義與實際頁面綁定
- dynamic import / lazy loading
- event 綁定點
- store / composable / hook 狀態更新
- API client 與回應映射

### 4. Queue / Scheduler / Worker
優先檢查：

- trigger 來源
- job / consumer 註冊
- payload 內容
- side effect
- retry / fallback / dead-letter 等邊界

### 5. Feature flag / Runtime config / Environment
優先檢查：

- 定義在哪裡
- 哪裡讀取
- 哪裡套用
- 目前是否有證據顯示在本流程中實際生效

若以上仍無法確認，應在輸出中明確寫出缺口，而不是假裝完整理解。

---

## 十、允許的最小驗證邊界

只有在以下情況，才可超出純閱讀進入最小驗證：

- 使用者明確要求驗證或除錯
- 不做驗證就無法區分兩條以上關鍵路徑
- 驗證行為為唯讀、低風險、可回溯
- 驗證的目的是消除理解阻擋，而不是直接完成修復

最小驗證可包含：

- 查看既有日誌
- 查看錯誤訊息
- 查看設定值
- 查看測試內容
- 查看唯讀命令輸出
- 在明確可控前提下執行不改變狀態的最小檢查

最小驗證不得擴張為：

- 修改原始碼
- 更新依賴
- 執行有副作用的流程
- 大範圍啟動系統
- 大範圍跑測試
- 直接完成修復或重構

---

## 十一、停止條件

滿足以下任一條件即可停止擴張：

- 已能直接回答使用者問題
- 已形成最小閉環
- 新閱讀內容只補背景，不改變結論
- 剩餘不確定性落在 runtime、資料內容、環境、外部系統、generated code 或未提供檔案上
- 已碰到清楚的系統邊界；再往外只會進入不相關模組

若最小閉環始終無法收斂，應改用以下收斂方式：

- 停在第一個可回答問題的系統邊界
- 拆成分段閉環（前端閉環、API 閉環、service→persistence 閉環）
- 明確交代停在哪裡、為什麼停、再往下需要什麼證據

---

## 十二、預設忽略項

除非與問題直接相關，否則不要優先展開：

- `node_modules`
- `dist`
- `build`
- `.next`
- `coverage`
- lock file
- 編譯輸出
- 大型 generated file
- 與入口無直接關聯的測試檔
- 只做 re-export 的 barrel file
- 純樣式檔
- 與當前流程無直接關聯的第三方原始碼

---

## 十三、模式判定

### `file`
理解單一檔案的角色、結構、依賴、關鍵邏輯、風險與下一步。

### `feature`
追蹤使用者可見功能、按鈕、端點、路由、錯誤流程、批次觸發鏈。

### `module`
理解一個有明確邊界的目錄、package、domain、功能群、workspace 或 service 的責任、核心檔案、典型流程、依賴、耦合、邊界。

判定原則：

- 問題核心是「這個檔案在做什麼」→ `file`
- 問題核心是「這個行為 / API / 錯誤怎麼發生」→ `feature`
- 問題核心是「這組檔案如何分工」→ `module`
- 若仍模糊，預設先用 `feature`

---

## 十四、輸出契約（最低標準）

除非使用者明確要求超短答，輸出至少要包含：

1. **範圍與入口**
2. **專案邊界**（若 relevant）
3. **實際閱讀 / 搜尋過的檔案**
4. **已確認**
5. **合理推論**
6. **待驗證**
7. **下一步**

### 每條核心結論的建議格式

- **[已確認]** 結論  
  - 證據：`檔案路徑:行號` / `檔案路徑#符號名` / 執行時證據簡述

- **[合理推論]** 結論  
  - 依據：哪些檔案 / 行為 / 設定支持這個推論  
  - 原因：為什麼目前只能推論，尚未完全確認

- **[待驗證]** 問題  
  - 缺口：缺哪個證據  
  - 下一步：應該讀哪個檔案、看哪個設定、檢查哪個執行時資訊

### 最低要求

- 重要結論不得無證據
- 搜尋命中不得直接當證據
- 若證據只到定義層，必須明說尚未確認生效
- 若缺乏行號，可用 `檔案路徑#方法名/類別名/段落名`
- 若證據來自日誌、錯誤訊息、執行輸出，必須明確標示為執行時證據

---

## 十五、何時直接回答、何時寫成 `.md`

### 直接在對話中回答
當符合以下任一情況時，可直接回答：

- 問題範圍窄
- 已能用局部最小閉環回答
- 結論、證據與下一步都很短
- 使用者明確要求直接回答

### 寫成 `.md`
符合以下情況時，預設整理成閱讀筆記：

- 跨檔案 / 跨層 / 跨模組
- 預期後續還會引用
- 使用者要求結構化筆記、模組總覽、功能追蹤筆記
- 直接塞進對話會影響可讀性

若未指定路徑，優先：

1. `docs/code-reading/`
2. `notes/code-reading/`
3. `docs/` 或 `notes/` 下合理子路徑
4. 若都沒有，使用 `docs/code-reading/`

檔名使用穩定的 kebab-case，並以 `-code-reading.md` 結尾，例如：

- `auth-login-code-reading.md`
- `order-service-code-reading.md`
- `pdf-viewer-rendering-code-reading.md`

## 十六、完成定義

這個 skill 何時算完成：

- 已有一條可追溯的最小閉環
- 已區分已確認 / 推論 / 待驗證
- 已說清楚證據在哪裡
- 已標出主要邊界與跨界點
- 已指出最有價值的下一步
- 未把猜測包裝成事實
- 未在閱讀階段滑入不必要的工程執行

若做不到完整理解，至少也要做到：

- **局部閉環清楚**
- **不確定性被誠實標示**
- **下一步足夠具體**

## 十七、最短版執行心法

若情況很複雜，永遠回到這七句：

1. 先找最接近問題的入口  
2. 只追到最小閉環  
3. 區分定義、註冊、生效  
4. 已確認一定要有證據  
5. 推論不能冒充事實  
6. 不確定就列待驗證  
7. 閱讀階段不要滑成實作階段
