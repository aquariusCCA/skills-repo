# 原始碼閱讀

使用這個 skill 來建立**可驗證、可追溯、可交接、可停止擴張**的既有程式碼理解。

核心目標不是「快速說明這段程式大概在做什麼」，而是回答：

- 程式目前**實際**做了什麼
- 哪些結論是**已確認**，證據在哪裡
- 哪些只是**合理推論**，推論依據是什麼
- 哪些仍屬於**待驗證**，缺口在哪裡
- 哪些證據彼此**衝突**，衝突點是什麼
- 下一步最值得閱讀、搜尋、縮邊界或最小驗證的是什麼

本 skill 的首要責任是：**降低誤讀，而不是提高說故事的流暢度。**

## 參考檔使用時機

主檔只保留每次觸發幾乎都需要的核心規則。需要更細節時再讀對應 reference：

- 入口不明、需要退避策略、核對清單、符號級閱讀模板、反誤讀規則或動態驗證時，讀 [references/deep-tracing.md](references/deep-tracing.md)
- 需要依框架決定預設閱讀順序時，讀 [references/framework-playbooks.md](references/framework-playbooks.md)
- 需要完整輸出格式、失敗分類或可直接套用的輸出骨架時，讀 [references/output-contracts.md](references/output-contracts.md)

不要預設把三份 reference 一次全部載入。

## 一、何時使用

以下情況適合使用這個 skill：

- 閱讀或解釋某個檔案、類別、方法、元件、端點、頁面、job、consumer、模組
- 追蹤某個功能、按鈕、路由、API、批次任務、資料流、呼叫鏈或副作用
- 在修改、除錯、重構、遷移、重寫前，先理解既有實作
- 產出可重複使用的閱讀筆記、功能筆記、模組筆記、定位報告
- 判斷下一步應該讀哪個檔案、哪個符號、哪一層邊界
- 釐清「看起來像是這樣」與「目前證據只能支持到哪裡」之間的落差

以下情況不要只靠這個 skill 直接回答：

- 主要需求是新架構設計、技術選型、完整重寫方案
- 使用者已明確要求直接修復、改寫、實作、部署、遷移
- 任務核心不是理解既有程式，而是產出新設計或新程式
- 使用者要的是產品決策、商業判斷，而非程式實際行為

這些情況下，本 skill 只負責「先理解既有程式」那一段。

## 二、核心任務定義

本 skill 要完成的不是「閱讀很多」，而是完成以下其中一種收斂：

1. **局部閉環**
   已能用一條局部、可追溯、可解釋的流程回答使用者當前問題
2. **分段閉環**
   已把前端、API、service、persistence、外部邊界拆成數段，各段內部清楚，但尚未跨段全部串完
3. **邊界定位**
   已定位到第一個真正阻斷理解的系統邊界，並明確指出再往下需要什麼證據

若三者都做不到，至少要做到：

- 誠實標示不確定性
- 指出失敗原因
- 提供下一步最小搜尋 / 最小閱讀 / 最小驗證建議

## 三、建議輸入

能提供越多越好，但不必每次完整提供：

- `goal`：閱讀目的，例如理解、定位、除錯前置、重構前置、驗證
- `entry`：入口檔案、類別、方法、路由、端點、UI 動作、錯誤位置、堆疊、log
- `mode`：`file`、`feature`、`module`、`debug-trace`；若未提供，自動判定
- `max_depth`：最多展開幾層呼叫鏈或依賴
- `constraints`：例如只看前端、只看後端、只看 SQL、只看本 service、限制輸出長度
- `expected_output`：例如直接回答、呼叫鏈、資料流、結構化筆記、下一步清單
- `project_boundary`：若為 monorepo、多服務、多工作區，指出只看哪個 app / service / package / workspace
- `confidence_target`：可選，若要求高可信度，必須更嚴格區分已確認與推論
- `runtime_available`：可選，是否允許做最小驗證，若未提供，預設僅靜態閱讀

## 四、規則優先級

若不同規則彼此衝突，依以下順序處理：

1. 使用者當前明確要求
2. 安全、唯讀、低副作用邊界
3. 本 skill 的明文規則
4. 專案既有慣例與目錄習慣
5. 一般框架慣例與命名推測

補充：

- 使用者要求「直接回答，不要建檔」時，優先直接回答
- 使用者要求與安全邊界衝突時，先完成閱讀結論，不在閱讀階段越界執行
- 專案慣例可參考，但**不得凌駕於直接證據**
- 框架慣例只能用來**縮小搜尋範圍**，不能直接當作事實

## 五、不可違反的核心原則

### 1. 先找入口，再追最小閉環
從最接近問題的入口開始。沒有明確入口時，先做輕量盤點，再選最保守起點。目標不是讀最多檔案，而是形成**最小可回答閉環**。

### 2. 最小閉環優先於大範圍掃描
只追與當前問題直接相關的呼叫鏈、資料流、狀態更新與副作用。若已能回答問題，就不要為了求全而盲目擴張。

### 3. 必須區分三層位置
閱讀時必須區分：

- **定義位置**：在哪裡宣告、描述、實作
- **註冊位置**：在哪裡被掛進系統、綁定、注入、映射
- **實際生效位置**：在當前問題中，真正被匹配、命中、執行的地方

只看到定義，不等於已確認生效。只看到註冊，也不等於本次流程一定會走到。

### 4. 證據分級是硬規則
所有結論只能落在四種狀態之一：

- **已確認**
- **合理推論**
- **待驗證**
- **證據衝突**

搜尋命中、命名直覺、框架慣例、過往經驗，都不能直接當已確認。

### 5. 先靜態閱讀，必要時才做最小驗證
預設先用唯讀方式建立理解。只有在**不做最小驗證就無法區分兩條以上關鍵路徑**時，才做唯讀、低風險、可回溯的最小驗證。

### 6. 優先縮邊界，不優先擴張閱讀量
monorepo、多服務、多工作區時，先確認這次請求屬於哪個 app、service、workspace、package。除非問題本身就是跨界流程，否則不要一開始就跨 service / workspace / app 擴張。

### 7. 多候選入口必須先排除，再展開
若找到多個入口、同名符號、多個實作、多條可能路徑，先列候選、說明理由、選最保守起點，其餘保留為備選。未排除主要替代路徑前，不得把單一路徑寫成已確認。

### 8. 閱讀階段不滑入工程執行
本 skill 的目的，是建立理解，不是直接修改原始碼、修 bug、重構、部署、安裝、更新資料。若需求已轉成工程執行，先完成閱讀結論，再切換到後續階段。

### 9. 衝突證據不可硬湊成單一路徑
若靜態程式、設定、測試、日誌、錯誤訊息彼此不一致，必須標示為**證據衝突**，指出衝突來源、可能原因，以及解衝突需要的下一步。

更細的退避策略、核對清單、反誤讀規則與動態驗證細節，讀 [references/deep-tracing.md](references/deep-tracing.md)。

## 六、證據分級標準

### A. 已確認
只有當前結論有**直接證據**支撐，且與當前問題有明確關聯時，才能標為已確認。

可接受證據包括：

- 實際讀到的程式碼路徑與符號呼叫
- 明確的路由 / 綁定 / 註冊 / 注入位置
- 明確的 SQL / query builder / XML 映射
- 明確的條件分支與命中條件
- 已存在的執行時證據：log、錯誤訊息、輸出、測試、trace

### B. 合理推論
有一定依據，但仍有缺口。例如找到 service 呼叫 repository，但未確認實際注入實作；或找到 route 與 component，但尚未確認當前頁面真的會觸發。

### C. 待驗證
目前缺乏關鍵證據，或者仍存在多條未排除路徑。例如多個同名實作尚未排除、feature flag 尚未確認、generated code 或外部服務邊界未提供。

### D. 證據衝突
不同證據來源彼此矛盾，例如程式碼顯示 A、log 顯示 B；設定顯示開啟功能，但測試與頁面行為不一致。

輸出衝突時必須寫清楚：

- 衝突來源
- 衝突內容
- 目前不能下定論的原因
- 解衝突需要的下一步

## 七、標準執行流程

除非使用者要求極短答，否則不要跳步。

### Step 0：確認目標
用一句話確認這次閱讀要回答什麼，以及是為了什麼。

### Step 1：確認邊界
先判定這次閱讀屬於單一檔案、單一功能流程、單一模組 / package / directory、單一 service / workspace / app，或跨界流程。若無法縮邊界，先交代目前最可能邊界與不確定點。

### Step 2：做最小盤點
只做有助於找入口的輕量盤點，例如 manifest、路由表、controller / API 定義、模組入口、框架設定、錯誤訊息、stack trace、符號搜尋與局部目錄樹。不要一開始掃整庫，也不要把盤點誤當成理解。

### Step 3：選入口
入口優先順序：

1. 使用者提供的錯誤位置、堆疊、異常訊息、UI 操作、API 名稱
2. 路由、controller、handler、event binding、UI action 綁定位置
3. service、hook、composable、use case
4. repository、mapper、DAO、API client、query builder
5. DTO、entity、schema、契約、設定檔
6. 測試、fixture、日誌、文件

若暫時找不到入口，讀 [references/deep-tracing.md](references/deep-tracing.md) 的退避策略。

### Step 4：讀入口，不要只看名稱
至少確認：

- 這裡是不是實際入口，還是只有定義 / 轉發 / re-export
- 它的輸入是什麼
- 它把控制權交給誰
- 它是否只是包裝層，而真正邏輯在下一層
- 目前看到的是定義、註冊，還是生效位置

### Step 5：沿關鍵路徑追到最小閉環
只追與目前問題直接相關的呼叫方 / 被呼叫方、import / export、state、API client、repository / SQL、DTO / schema、route / event、config / feature flag、測試 / log / 樣例與外部契約邊界。

### Step 6：做收斂檢查
在宣稱已形成閉環前，至少檢查：

- 是否已看到入口
- 是否已看到主要轉交點
- 是否已看到副作用或輸出位置
- 是否已確認至少一個主要分支條件
- 是否已排除至少一條主要替代路徑
- 是否已區分定義 / 註冊 / 生效

若以上任兩項答不出，不應過早宣稱已收斂。

### Step 7：對每段關鍵結論做證據分級
每條重要結論都必須標示為：已確認、合理推論、待驗證、或證據衝突，並補上證據、缺口或衝突來源。

### Step 8：檢查是否該停止
出現以下任一情況即可停止擴張：

- 已能直接回答使用者問題
- 已形成最小閉環或分段閉環
- 新展開檔案不再改變結論
- 剩餘不確定性只落在 runtime、資料內容、外部系統、環境、generated code 或未提供檔案上
- 再展開只會跨到不相關模組或其他 service / workspace
- 成本開始超過資訊增益

### Step 9：輸出結果
至少交代：

- 這次要回答的問題
- 範圍與入口
- 專案邊界
- 實際讀過 / 搜過哪些檔案
- 閱讀到的主路徑
- 已確認
- 合理推論
- 待驗證
- 證據衝突（若有）
- 目前停在哪裡
- 下一步

更完整的輸出格式與失敗輸出規則，讀 [references/output-contracts.md](references/output-contracts.md)。

## 八、允許的最小驗證邊界

只有在以下情況，才可超出純閱讀進入最小驗證：

- 使用者明確要求驗證或除錯
- 不做驗證就無法區分兩條以上關鍵路徑
- 驗證行為為唯讀、低風險、可回溯
- 驗證的目的是消除理解阻擋，而不是直接完成修復

最小驗證可包含：

- 查看既有日誌
- 查看錯誤訊息
- 查看設定值
- 查看測試內容
- 查看唯讀命令輸出
- 在可控前提下執行不改變狀態的最小檢查
- 讀取 trace、debug 輸出、SQL log、network log

最小驗證不得擴張為：

- 修改原始碼
- 更新依賴
- 執行有副作用的流程
- 大範圍啟動系統
- 大範圍跑測試
- 直接完成修復或重構
- 將閱讀任務偷偷滑成工程任務

更細的動態行為驗證策略，讀 [references/deep-tracing.md](references/deep-tracing.md)。

## 九、停止條件

滿足以下任一條件即可停止擴張：

- 已能直接回答使用者問題
- 已形成最小閉環或分段閉環
- 新閱讀內容只補背景，不改變結論
- 剩餘不確定性落在 runtime、資料內容、環境、外部系統、generated code 或未提供檔案上
- 已碰到清楚的系統邊界；再往外只會進入不相關模組
- 已有足夠證據支持下一步工程行動

若最小閉環始終無法收斂：

- 停在第一個可回答問題的系統邊界
- 拆成分段閉環
- 輸出目前只能確認到哪裡
- 明確交代停在哪裡、為什麼停、再往下需要什麼證據

## 十、預設忽略項

除非與問題直接相關，否則不要優先展開：

- `node_modules`
- `dist`
- `build`
- `.next`
- `coverage`
- lock file
- 編譯輸出
- 大型 generated file
- 與入口無直接關聯的測試檔
- 只做 re-export 的 barrel file
- 純樣式檔
- 與當前流程無直接關聯的第三方原始碼

若問題本身與 build、codegen、bundling、transpile、polyfill 有關，才把這些項目納入邊界。

## 十一、模式判定

### `file`
理解單一檔案的角色、結構、依賴、關鍵邏輯、風險與下一步。

### `feature`
追蹤使用者可見功能、按鈕、端點、路由、錯誤流程、批次觸發鏈。

### `module`
理解一個有明確邊界的目錄、package、domain、功能群、workspace 或 service 的責任、核心檔案、典型流程、依賴、耦合、邊界。

### `debug-trace`
當使用者給的是錯誤訊息、堆疊、log、異常行為時，從故障現象往回追到最小閉環。

判定原則：

- 問題核心是「這個檔案在做什麼」→ `file`
- 問題核心是「這個行為 / API / 錯誤怎麼發生」→ `feature`
- 問題核心是「這組檔案如何分工」→ `module`
- 問題核心是「這個錯誤從哪裡來」→ `debug-trace`
- 若仍模糊，預設先用 `feature`

需要依技術棧決定預設閱讀順序時，讀 [references/framework-playbooks.md](references/framework-playbooks.md)。

## 十二、輸出契約（最低標準）

除非使用者明確要求超短答，輸出至少要包含：

1. **這次要回答的問題**
2. **範圍與入口**
3. **專案邊界**
4. **實際閱讀 / 搜尋過的檔案**
5. **閱讀到的主路徑**
6. **已確認**
7. **合理推論**
8. **待驗證**
9. **證據衝突**（若有）
10. **目前停在哪裡**
11. **下一步**

最低要求：

- 重要結論不得無證據
- 搜尋命中不得直接當證據
- 若證據只到定義層，必須明說尚未確認生效
- 若缺乏行號，可用 `檔案路徑#方法名/類別名/段落名`
- 若證據來自日誌、錯誤訊息、執行輸出，必須標示為執行時證據
- 若存在主要替代路徑未排除，不得把結論寫成唯一事實

詳細格式、失敗輸出契約與可直接套用骨架，讀 [references/output-contracts.md](references/output-contracts.md)。

## 十三、何時直接回答、何時寫成 `.md`

### 直接在對話中回答
符合以下任一情況時，可直接回答：

- 問題範圍窄
- 已能用局部最小閉環回答
- 結論、證據與下一步都很短
- 使用者明確要求直接回答

### 寫成 `.md`
符合以下情況時，預設整理成閱讀筆記：

- 跨檔案 / 跨層 / 跨模組
- 預期後續還會引用
- 使用者要求結構化筆記、模組總覽、功能追蹤筆記
- 直接塞進對話會影響可讀性
- 需要保留證據鏈與未解決缺口

若未指定路徑，優先：

1. `docs/code-reading/`
2. `notes/code-reading/`
3. `docs/` 或 `notes/` 下合理子路徑
4. 若都沒有，使用 `docs/code-reading/`

檔名使用穩定的 kebab-case，並以 `-code-reading.md` 結尾，例如：

- `auth-login-code-reading.md`
- `order-service-code-reading.md`

## 十四、完成定義

這個 skill 何時算完成：

- 已有一條可追溯的最小閉環，或至少一組分段閉環
- 已區分已確認 / 推論 / 待驗證 / 證據衝突
- 已說清楚證據在哪裡
- 已標出主要邊界與跨界點
- 已指出最有價值的下一步
- 已排除至少一條主要替代路徑，或誠實說明尚未排除
- 未把猜測包裝成事實
- 未在閱讀階段滑入不必要的工程執行

若做不到完整理解，至少也要做到：

- **局部閉環清楚**
- **不確定性被誠實標示**
- **失敗原因被正確分類**
- **下一步足夠具體**

## 十五、最短版執行心法

若情況很複雜，永遠回到這十句：

1. 先找最接近問題的入口
2. 只追到最小閉環
3. 區分定義、註冊、生效
4. 已確認一定要有證據
5. 推論不能冒充事實
6. 有替代路徑就先排除
7. 不確定就列待驗證
8. 證據打架就列衝突
9. 閱讀階段不要滑成實作階段
10. 找不到入口就誠實停在邊界

## 十六、最後約束

這個 skill 的成功標準不是「讀起來很像懂了」，而是：

- 使用者能知道你**確定了什麼**
- 使用者能知道你**還沒確定什麼**
- 使用者能知道你**為什麼這樣判斷**
- 使用者能知道你**接下來該看哪裡**
- 後續接手的人能沿著你的證據鏈繼續往下做

只要做不到以上任一點，就不應假裝已經完成理解。
