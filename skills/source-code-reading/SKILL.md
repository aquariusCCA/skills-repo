# 原始碼閱讀

使用這個 skill 來建立有證據支撐的既有程式碼理解。目標是說明程式目前實際做了什麼、證據在哪裡、還有哪些不確定，以及下一步最值得閱讀什麼。

這個 skill 以先理解為主。如果使用者要求修復、重構、遷移或實作，先用這個 skill 建立相關上下文，再繼續完成使用者要求的工程工作。

## 何時使用

以下請求適合使用這個 skill：

- 閱讀或解釋某個檔案、類別、方法、元件、API、頁面、工作或模組。
- 追蹤某個功能、按鈕、路由、端點、批次工作、呼叫鏈或資料流。
- 在修改前理解陌生或舊有專案。
- 產出可重複使用的模組或功能筆記。
- 為聚焦任務找出下一步應該閱讀的檔案。
- 在除錯、重構、遷移或重寫前，先建立局部流程或模組邊界的可驗證理解。

如果使用者的主要需求是架構設計、技術選型、大範圍重寫規劃或完整實作，不要只用這個 skill 回答。這些情況下，只把它用於程式碼理解的部分。

## 建議輸入

能提供越多越好，但不必每次都完整提供。

- `goal`：這次閱讀是為了什麼，例如理解、除錯前置、重構前置、功能追蹤。
- `entry`：入口檔案、類別、方法、路由、端點、模組、UI 動作或錯誤位置。
- `mode`：`file`、`feature`、`module`；若未提供，依請求自動判定。
- `max_depth`：可選。限制最多展開幾層依賴或呼叫鏈；若未提供，依最小閉環原則決定展開範圍。
- `constraints`：可選。像是只看前端、只看後端、只看 SQL、限制輸出長度、限制時間。
- `expected_output`：可選。可指定回答形式，例如直接回答、精簡筆記、結構化筆記、呼叫鏈、資料流、下一步閱讀清單，或指定輸出到特定 `.md` 檔案。若未指定，依「輸出指引」決定。
- `project_boundary`：可選。若專案是 monorepo、多服務或多工作區，指出目前只看哪個 app、service、package、workspace 或子系統。

## 規則優先級

若不同規則、使用者要求或專案慣例互相衝突，依以下優先順序處理：

1. 使用者當前明確要求。
2. 安全、唯讀與低副作用邊界。
3. 本 skill 的明文規則。
4. 專案既有慣例與目錄習慣。
5. 一般框架慣例或命名推測。

補充說明：

- 若使用者要求「直接回答，不要建檔」，即使任務較長，也應優先直接在對話中輸出。
- 若專案慣例與本 skill 衝突，例如慣例把分析筆記寫在特殊位置，但 skill 明確限制輸出路徑，先依 skill；除非使用者另有指定。
- 若使用者要求與安全邊界衝突，例如要求大範圍執行或修改，先完成閱讀結論，再切到後續工程工作，不要在閱讀階段直接越界。

## 核心規則

- 盡可能從具體入口開始：`entry`、檔案、類別、方法、功能名稱、路由、端點、模組、UI 動作或錯誤位置。
- 如果沒有提供入口，先做輕量盤點：用 `rg --files` 列出檔案，檢查明顯的 manifest、路由、設定檔、workspace 邊界與模組入口，再選擇保守的起點；只有在任務仍然模糊、且無法做出保守選擇時才提問。
- 如果找到多個合理入口，先列出 2 到 4 個候選入口與判斷依據，選擇最保守、最接近使用者問題、最容易形成最小閉環的入口開始；其餘列為備選路徑，不要一開始就全部展開。
- 只展開與目標直接相關的依賴、呼叫方與被呼叫方。若使用者提供 `max_depth` 就依其限制；否則不要固定只追一層，而是追到足以解釋問題的最小閉環為止。
- 「最小閉環」是指：已能用一條可解釋的局部流程回答使用者問題，而不是停在半路。常見例子：
  - 前端互動：路由 / 事件 -> 元件 / store -> API -> 狀態更新 / 渲染。
  - 後端請求：controller -> service -> repository / mapper -> SQL / 外部 API -> response。
  - 批次工作：trigger -> job -> service -> persistence / side effect。
- 若專案是 monorepo、多服務或多工作區結構，先確認這次請求所屬的 app、service、package、workspace 或子系統邊界。除非問題本身就是跨服務流程，否則不要跨 workspace、跨 service 或跨 app 擴張閱讀範圍。
- 若問題本身涉及跨前後端、跨服務或跨工作區流程，先明確標出跨界點，再分段追蹤每一段最小閉環，不要直接把整個系統攤平。
- 區分證據狀態：已確認、合理推論、待驗證。
- 證據不是附註，而是結論的一部分。凡標示為「已確認」的行為，都必須附上至少一個直接證據。
- 「合理推論」必須附上所依據的證據，並簡短說明推論成立的原因；不得把推論寫成已確認事實。
- 「待驗證」必須明確指出目前缺少的證據、阻擋點，以及下一步要讀取或檢查的檔案、命令或執行時資訊。
- 若某個結論目前沒有直接證據，必須明確標示為推論或待驗證，不得以肯定語氣陳述。
- 證據可來自已開啟檔案、搜尋命中、命令輸出或執行時資訊；引用時應盡可能提供可追溯的位置。
- 不要宣稱已閱讀沒有實際開啟或搜尋過的檔案。
- 不要從單一檔案、單一 controller、單一 component 或單一命名慣例推斷完整系統設計。
- 重要結論優先附上可追溯的證據參考；若證據來自檔案，應優先附上檔案與行號。
- 輸出形式、落檔位置、最低欄位與模板使用，統一依「輸出指引」辦理；不要在其他段落重複定義。
- 若目前已有足夠資訊可開始最小閉環閱讀，優先直接開始；不要為了補齊非必要欄位而中斷任務提問。
- 若必須提問，問題應最小化，只問會直接影響入口選擇、專案邊界或輸出形式的關鍵資訊。

## 非理想情況與降級策略

若專案不標準、工具受限或資訊不足，不要硬套標準流程；應採保守降級策略，並在輸出中說明限制。

- 若無法使用 `rg --files` 或等價搜尋工具，改用可用的列檔、目錄瀏覽或框架設定檔作為輕量盤點來源。
- 若專案缺少明顯 manifest、路由表或入口設定，先從使用者提供的檔名、錯誤訊息、模組名稱、API 名稱或 UI 動作逆推可能入口。
- 若專案過大，例如巨型 monorepo、多層 generated code、vendor 內容龐大，先縮到單一 workspace、app、service 或 package；必要時先輸出邊界判定，而不是直接展開整庫。
- 若遇到 binary、minified、編譯產物、巨大 generated file 或極難閱讀的 codegen 輸出，除非與問題直接相關，否則應略過並註明原因。
- 若搜尋結果太多且無法一次形成保守入口，先列出候選入口與各自理由，不要一次展開所有命中。
- 若目前可得資訊只足以建立局部結論，應先輸出局部最小閉環與缺口，而不是為了追求完整性無限制擴張。

## 動態行為與靜態閱讀上限

以下類型的行為，常常無法只靠靜態閱讀完全確認；除非已有直接證據，否則應降級為合理推論或待驗證：

- 依賴 runtime config、environment variable、profile、feature flag 或部署參數的分支。
- 依賴 reflection、annotation 掃描、AOP、DI 自動裝配、慣例式註冊或 plugin 註冊的行為。
- 依賴 dynamic import、lazy route、runtime route registration、code splitting 或 remote module 的前端流程。
- 依賴 MyBatis 動態 SQL、ORM 映射、stored procedure、資料庫 trigger、view 或資料內容本身的邏輯。
- 依賴 queue、排程、非同步重試、事件總線、外部系統 callback 或跨服務契約的流程。
- 依賴 codegen、schema 產物、OpenAPI / gRPC 產物或其他生成鏈的行為。

遇到這些情況時：

- 優先寫清楚哪一段是靜態可見、哪一段只是合理推論。
- 若需要最小驗證，優先使用唯讀、低成本、低副作用方式。
- 若無法驗證，不要假裝完整理解；應明確列入「待驗證」。

## 閱讀階段邊界

這個 skill 的主要目的，是建立可驗證的程式碼理解，而不是直接完成修復、重構、遷移或新實作。閱讀階段應以最小必要行動取得足夠上下文，避免滑入不必要的工程執行。

閱讀階段預設允許的行為包括：

- 讀取檔案、搜尋符號、追蹤呼叫鏈、整理模組邊界與資料流。
- 做輕量專案掃描，例如檢查 manifest、workspace 設定、路由表、模組入口、設定檔與目錄結構。
- 使用不改變專案內容的唯讀命令取得線索，例如列檔、搜尋、檢視設定、檢視既有日誌、檢視錯誤訊息或檢視測試檔內容。
- 在必要時整理假設、缺口與下一步驗證方向，但必須清楚標示為合理推論或待驗證。
- 在符合「輸出指引」的情況下，新增或更新閱讀筆記 `.md` 檔案；此類筆記僅限於分析輸出，不視為修改業務程式、設定或既有產品文件。

閱讀階段預設不做的行為包括：

- 修改原始碼、設定檔、測試、資料、生成產物，或不相干的既有業務文件。
- 主動實作新功能、修補 bug、重構既有程式、調整架構或替使用者完成工程變更。
- 執行具有副作用、寫入、刪除、遷移、部署、發布、安裝或更新性質的命令。
- 在沒有明確需要的情況下啟動整個系統、完整建置、完整測試或大範圍執行流程。
- 把閱讀階段得到的局部理解，直接包裝成完整修復方案或已驗證的最終實作。

只有在以下情況，才可超出純閱讀行為做最小驗證：

- 使用者明確要求驗證、除錯或執行。
- 不做最小驗證就無法區分兩個以上關鍵路徑，且這個驗證行為是唯讀、低風險、可回溯的。
- 驗證本身是為了消除目前阻擋理解的關鍵不確定性，而不是為了直接完成實作。

若進行最小驗證，應遵守以下限制：

- 優先選擇唯讀、低成本、低副作用的方式。
- 只驗證與當前問題直接相關的最小範圍。
- 在輸出中明確說明做了什麼驗證、驗證結果是什麼，以及哪些部分仍未驗證。
- 不得把驗證動作擴張成完整修復、完整測試或完整實作。

若使用者的真正需求已明確轉為修復、重構、遷移或實作，應先用本 skill 產出足夠的閱讀結論，再切換到後續工程工作；不要把兩個階段混寫成同一份閱讀筆記。

## 停止條件

滿足以下任一條件即可停止擴張閱讀範圍：

- 已能直接回答使用者問題。
- 已形成足以解釋問題的最小閉環。
- 新展開的檔案只補充背景，沒有實質改變目前結論。
- 剩餘不確定性已明確落在執行時資訊、資料庫內容、環境設定、外部系統、生成碼或未提供的檔案上。
- 已碰到清楚的專案邊界；若再擴張只會進入不相關模組、其他服務或其他工作區。

若最小閉環遲遲無法收斂，應採以下收斂方式，而不是無限擴張：

- 優先停在第一個足以回答使用者問題的系統邊界，例如前端 API 呼叫點、後端 service 邊界、repository/mapper 邊界或外部系統交界點。
- 把流程拆成分段閉環，例如前端閉環、API 閉環、service 到 persistence 閉環。
- 明確說明目前停在哪一段、為什麼停在這裡，以及再往下需要什麼額外證據。

## 預設忽略

除非與目前問題直接相關，否則不要優先展開：

- `node_modules`、`dist`、`build`、`.next`、`coverage`。
- lock file、編譯輸出與其他大型生成檔。
- 與目前入口無直接關聯的測試檔。
- 只做 re-export 的 barrel files。
- 純樣式檔，除非問題與 UI 呈現、class 切換或樣式條件有關。
- 與當前流程無直接關聯的第三方函式庫原始碼。
- 大型自動生成檔、schema 產物或 codegen 輸出，除非問題本身與生成流程或生成結果有關。

## 預設模式

根據使用者請求選擇模式。若使用者指定模式，就照指定模式執行。

- `file`：理解單一檔案的角色、結構、依賴、關鍵邏輯、風險與下一步閱讀方向。
- `feature`：追蹤使用者可見功能、端點、按鈕、路由、批次工作或跨檔流程。
- `module`：以一個有明確邊界的目錄、package、domain、功能群或子系統為閱讀單位，根據實際讀過的檔案整理模組責任、主要檔案、典型流程、依賴、耦合與缺口。

如果使用者只提供檔案路徑，預設使用 `file`。
如果使用者提供功能名稱加入口，預設使用 `feature`。
如果使用者提供目錄、package、workspace、service 或模組並要求總覽，預設使用 `module`。

若請求同時帶有檔案、功能與模組訊號，依以下原則判定主模式：

- 問題核心是「這個檔案本身在做什麼」：優先 `file`。
- 問題核心是「某個行為 / 按鈕 / API / 錯誤為什麼發生」：優先 `feature`，即使使用者也提供了檔案路徑。
- 問題核心是「這組檔案如何分工、模組責任如何切」：優先 `module`。
- 若仍模糊，預設先用 `feature`，因為它通常最接近使用者的真實問題，再在輸出中說明目前的入口與邊界。

### `module` 模式的意思

`module` 不是要求讀完整個專案，也不是把某個目錄底下每個檔案逐一摘要。它適合用在使用者想理解「一組相關檔案共同負責什麼」時：先從目錄、package、公開入口或代表性流程做保守掃描，再閱讀足以說明模組責任與典型流程的檔案。

適合使用 `module` 的情況：

- 使用者提供模組目錄，想知道它的責任與主要檔案，例如 `src/modules/auth`。
- 使用者提供後端 package，想理解 controller、service、repository / mapper、DTO/entity 的責任分布，例如 `src/main/java/com/example/order`。
- 使用者準備重構某個功能群，想先知道目前耦合、風險與下一步閱讀點，例如 `src/features/pdf-viewer`。
- 使用者想建立可重用筆記，而不是只回答某個按鈕、某個 API 或某個函式怎麼跑。
- 在 monorepo 中，使用者想理解某個 workspace、app 或 service 的局部責任，而不是整個倉庫。

不適合使用 `module` 的情況：

- 只有單一檔案要解釋：使用 `file`。
- 只追一個端點、按鈕、路由、錯誤流程或批次觸發鏈：使用 `feature`。
- 問題是完整系統架構、技術選型或大範圍重寫：不要只用這個 skill 回答；先把範圍縮到可讀的模組或流程。
- 使用者要求「把整個目錄所有檔案都摘要」：先確認這是清單盤點需求，否則仍以代表性檔案與最小閉環為主。

`module` 範例：

- `entry: src/modules/auth`：整理登入、登出、session 狀態、API client、route guard 或 store 的責任分布與典型流程。
- `entry: src/main/java/com/example/order`：整理 order package 中 controller、service、mapper / repository、DTO/entity 如何分工，並列出常見 request 到 persistence 的流程。
- `entry: src/features/pdf-viewer`：整理 PDF viewer 相關元件、composable / hook、store、rendering 或 worker 邏輯的邊界與耦合風險。
- `entry: jobs/invoice`：整理 invoice batch 的 scheduler / job / service / persistence / side effect 邊界與資料更新點。
- `entry: apps/web/src/features/profile`：整理 monorepo 中 web app 的 profile 功能責任、依賴與代表性流程。
- `entry: services/payment`：整理 payment service 的對外入口、主要流程與與其他服務的交界點。
- `entry: src/shared/form`：整理 shared form 模組提供哪些可重用元件或工具、依賴哪些驗證邏輯，以及哪些呼叫方最能代表它的使用方式。

## 入口與邊界判定

在正式展開閱讀前，先做保守判定：

- **單一明確入口**：直接從該入口開始。
- **沒有明確入口**：先做輕量掃描，從最可能形成最小閉環的入口開始。
- **有多個候選入口**：先列出候選入口與判斷依據，選最保守的起點；其餘保留為備選，不同步擴張。
- **monorepo / 多服務 / 多工作區**：先確認這次請求屬於哪個 app、workspace、package 或 service，再開始閱讀。
- **跨界流程**：先標出每個跨界點，例如前端 -> API、gateway -> service、service -> queue、service -> DB，再分段追蹤。
- **仍然模糊且無法保守決策**：才提問，而且只問會改變入口、邊界或輸出形式的最小問題。

## 閱讀流程

1. 用一句話確認目標：這次要理解什麼，以及為了什麼目的。
2. 整理已知輸入：`goal`、`entry`、`mode`、`max_depth`、`constraints`、`expected_output`、`project_boundary`。
3. 確認專案邊界與入口；若有多個候選入口，先列候選與判斷依據，再選擇起點。
4. 只在需要時做輕量專案掃描：manifest、workspace 設定、路由表、模組目錄、框架設定或符號搜尋。
5. 先閱讀入口檔案。
6. 只追蹤相關的呼叫方、被呼叫方、import、路由、API 呼叫、SQL / mapper、DTO / entity 定義、設定或 UI 事件，並追到形成最小閉環或達到 `max_depth` 為止。
7. 閱讀時追蹤證據狀態：
   - 已確認：可從已開啟檔案或可靠命令輸出直接看見。
   - 合理推論：根據命名、框架慣例、目錄結構、部分呼叫關係或靜態線索推論。
   - 待驗證：重要結論受未讀檔案、缺少執行時脈絡、隱藏 SQL、生成碼、環境設定、多種可能流程、動態註冊或跨服務邊界阻擋。
8. 若某段流程跨出當前邊界，先標明跨界點，再判斷是否有必要追下一段最小閉環。
9. 依使用者要求的格式回答；若未指定，依「輸出指引」決定輸出形式、是否落檔，以及是否使用 `references/output-templates.md`。
10. 除非回答本來就需要非常簡短，結尾應指出最高價值的下一個閱讀目標或下一步行動。

## 技術線索

優先依照專案實際檔案與慣例判斷。以下只是提醒，不是固定技術棧清單。

- Java web / Spring / Spring Boot：檢查 controller-service-repository 或 mapper 流程、DI、DTO/entity 流動、驗證、交易邊界、例外處理與設定。
- Hibernate / MyBatis / SQL-heavy 專案：檢查 DAO/mapper XML 或註解、HQL/SQL、stored procedure、result mapping、交易邊界與 entity/DTO mapping。
- Vue / React / 前端專案：檢查路由、元件邊界、props/events、store/composables/hooks、API client、狀態轉換、表單處理與渲染條件。
- jQuery / 舊式前端：檢查 DOM 綁定、初始化流程、全域狀態、AJAX 呼叫、工具函式耦合、動態 HTML 生成與表單參數組裝。
- Batch / Scheduler / Worker：檢查 trigger、job、queue consumer、service、副作用、重試邏輯、持久化與外部整合點。
- Monorepo / 多服務：檢查 workspace 設定、app/service 邊界、shared package、跨服務 API / queue / schema 契約與部署切面。
- 未知技術棧：先檢查 manifest、workspace 設定與入口/設定檔，再套用假設。

## 證據格式與可信度要求

除非使用者明確要求極短回答，否則輸出中的結論應盡量符合以下最低標準：

- 每個「已確認」結論至少附 1 個直接證據。
- 每個「合理推論」至少附 1 個依據，並簡短說明為何形成該推論。
- 每個「待驗證」都要寫明缺少的是什麼證據，以及下一步該看哪裡。
- 搜尋命中但未開檔，只能列為線索或候選入口，不可列為已確認。
- 若證據來自檔案，優先使用 `檔案路徑:行號`；若無法穩定取得行號，使用 `檔案路徑#符號名/方法名/段落名`。
- 若證據來自命令輸出，應簡短描述是哪個命令或哪類輸出，不要假裝它來自檔案。
- 若證據來自執行時資訊、日誌或錯誤訊息，應明確標示這是執行時證據，而非靜態碼證據。

## 輸出指引

本 skill 的輸出相關規則，統一以本節為準。其他段落若提到輸出形式、落檔位置、最低欄位或模板使用，只做簡短引用，不重複定義。

### 預設輸出原則

預設輸出為 Markdown 格式閱讀筆記。輸出長度與結構應符合使用者請求；窄問題用精簡 Markdown，較廣、跨檔案、可重用或明確要求結構化整理時，使用結構化 Markdown 筆記。

若使用者已指定輸出形式、輸出路徑、檔名或長度限制，優先依使用者要求處理。若未指定，依本節規則決定。

### 何時直接在對話輸出

符合以下任一情況時，可以直接在對話中輸出 Markdown：

- 問題範圍窄，已能用局部最小閉環直接回答。
- 使用者只要短答案、一次性判斷或快速釐清。
- 閱讀結果主要是單一結論、少量證據與明確下一步，沒有必要落成可重用筆記。
- 使用者明確要求直接回答，不需建立檔案。

直接在對話輸出時，除非使用者明確要求超短答，仍應保留最基本的可驗證資訊，不要只給結論不附依據。

### 何時寫入 `.md`

符合以下任一情況時，預設寫入 `.md` 檔案，並在回覆中明確列出檔案路徑：

- 任務較長、跨檔案、跨模組或跨層流程。
- 內容適合後續重讀、延伸或作為修改前上下文。
- 使用者要求結構化筆記、閱讀報告、模組總覽、功能追蹤筆記或可重用整理。
- 回答若直接塞在對話中，會影響可讀性或後續追蹤。
- 使用者明確要求建立檔案或指定輸出到某個 `.md` 路徑。

若使用者明確指定輸出路徑或檔名，照指定位置寫入；若未指定，依下列路徑優先順序決定。

### 落檔操作細則

是否落檔，依以下規則決定：

- 使用者明確要求輸出到檔案、指定路徑、指定檔名，或要求建立可重用筆記時，必須落檔。
- 若任務屬於較長、結構化、可重用、跨檔案、跨步驟或預期後續會持續引用的閱讀筆記，預設落檔。
- 若任務只是窄問題、短答案、一次性判斷，且直接在對話中回答已足夠，預設不落檔。
- 閱讀筆記的新增或更新，僅限本 skill 指定的輸出位置或使用者指定位置；不得藉由「落檔」之名修改不相干的既有產品文件或規格文件。
- 不得在未告知使用者的情況下隱性建立檔案；若有落檔，回覆中必須明確列出建立或更新的檔案路徑。

落檔路徑依「輸出路徑優先順序」決定；若選定的子目錄不存在，可建立必要目錄。單一任務中應使用單一路徑，不要把同一份閱讀筆記拆散到多個資料夾。

檔名應以本次閱讀的範圍或主題命名，使用穩定、可辨識的 kebab-case，並以 `-code-reading.md` 結尾。優先使用功能、模組、領域或入口名稱；避免使用過度泛化的名稱，如 `index.md`、`main.md`、`note.md`。例如：
- `auth-login-code-reading.md`
- `order-service-code-reading.md`
- `pdf-viewer-rendering-code-reading.md`

若 `entry` 是具體檔案，檔名應取自最有辨識度的功能名或模組名，而不是機械地沿用 `index`、`page`、`main` 這類通用檔名。若範圍跨 workspace、app 或 service，可在檔名前加上邊界前綴，例如 `web-profile-code-reading.md`、`payment-refund-code-reading.md`。

若目標檔案已存在，且本次請求與既有筆記屬於相同範圍、相同主題或同一閱讀任務的延伸，預設更新既有檔案，而不是新建重複檔案。只有在以下情況才新建檔案：
- 使用者明確要求保留歷史版本或另存新檔。
- 本次請求的範圍、目標或專案邊界已明顯改變，不再適合視為同一份筆記。
- 既有檔名已被不相關內容占用。

若需要新建同主題後續版本，優先使用穩定且可排序的後綴，例如 `-v2`、`-v3`；不要任意使用難以理解的檔名。

回覆中應明確說明本次是：
- 直接在對話中回答，未落檔；
- 建立新檔；
- 更新既有檔案；
並列出最終檔案路徑。

### 輸出路徑優先順序

若使用者沒有指定輸出路徑，輸出 `.md` 檔案時依以下優先順序選擇位置：

1. 專案既有的 `docs/code-reading/`
2. 專案既有的 `notes/code-reading/`
3. 專案既有的 `docs/` 或 `notes/` 下合理子路徑
4. 若都沒有，使用 `docs/code-reading/`

檔名應清楚反映閱讀主題、模組或流程，避免使用過度籠統的名稱。

### 最低欄位

除非使用者明確要求超短答，輸出至少應包含以下欄位：

- 範圍與入口
- 實際閱讀或搜尋過的檔案
- 已確認行為
- 待驗證項目
- 下一步閱讀或下一步行動

若內容非常短，也可以用精簡形式表達，但不得省略「已確認」與「待驗證」的區分。

### 完整結構化欄位

當任務較廣、需要可重用筆記，或使用者要求結構化整理時，輸出應盡量包含以下內容：

- 範圍與入口
- 實際閱讀或搜尋過的檔案
- 已確認行為
- 合理推論，並清楚標示
- 待驗證項目
- 與請求相關的依賴、呼叫方、被呼叫方或資料流
- 相關時才列出的風險、可疑區域或邊界限制
- 下一步檔案或下一步行動，以及原因

欄位不是越多越好；若某些欄位不適用、資訊不足，或加入後只會破壞閱讀流暢度，可以省略或合併，但不能偽造完整性。

### 何時使用 `references/output-templates.md`

只有在目前請求適合完整筆記格式時，才讀取 `references/output-templates.md` 的詳細模板，例如：

- 使用者要求完整閱讀筆記或模組總覽。
- 任務涉及多檔案、多段流程或跨邊界追蹤。
- 內容預期會被反覆查閱或交接使用。
- 需要較穩定的輸出骨架來維持一致性。

`references/output-templates.md` 是模板，不是硬性格式。實際輸出應依任務內容調整、合併或省略局部欄位，避免為了套模板而破壞理解順序或閱讀流暢度。
