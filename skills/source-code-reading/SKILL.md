# 原始碼閱讀

使用這個 skill 來建立有證據支撐的既有程式碼理解。目標是說明程式目前實際做了什麼、證據在哪裡、還有哪些不確定，以及下一步最值得閱讀什麼。

這個 skill 以先理解為主。如果使用者要求修復、重構、遷移或實作，先用這個 skill 建立相關上下文，再繼續完成使用者要求的工程工作。

## 何時使用

以下請求適合使用這個 skill：

- 閱讀或解釋某個檔案、類別、方法、元件、API、頁面、工作或模組。
- 追蹤某個功能、按鈕、路由、端點、批次工作、呼叫鏈或資料流。
- 在修改前理解陌生或舊有專案。
- 產出可重複使用的模組或功能筆記。
- 為聚焦任務找出下一步應該閱讀的檔案。

如果使用者的主要需求是架構設計、技術選型、大範圍重寫規劃或完整實作，不要只用這個 skill 回答。這些情況下，只把它用於程式碼理解的部分。

## 建議輸入

能提供越多越好，但不必每次都完整提供。

- `goal`：這次閱讀是為了什麼，例如理解、除錯前置、重構前置、功能追蹤。
- `entry`：入口檔案、類別、方法、路由、端點、模組、UI 動作或錯誤位置。
- `mode`：`file`、`feature`、`module`；若未提供，依請求自動判定。
- `max_depth`：可選。限制最多展開幾層依賴或呼叫鏈；若未提供，依最小閉環原則決定展開範圍。
- `constraints`：可選。像是只看前端、只看後端、只看 SQL、限制輸出長度、限制時間。
- `expected_output`：可選。直接回答、結構化筆記、呼叫鏈、資料流、下一步閱讀清單。

## 核心規則

- 盡可能從具體入口開始：`entry`、檔案、類別、方法、功能名稱、路由、端點、模組、UI 動作或錯誤位置。
- 如果沒有提供入口，先做輕量盤點：用 `rg --files` 列出檔案，檢查明顯的 manifest、路由與設定檔，再選擇保守的起點；只有任務仍然模糊時才提問。
- 只展開與目標直接相關的依賴、呼叫方與被呼叫方。若使用者提供 `max_depth` 就依其限制；否則不要固定只追一層，而是追到足以解釋問題的最小閉環為止。
- 「最小閉環」是指：已能用一條可解釋的局部流程回答使用者問題，而不是停在半路。常見例子：
  - 前端互動：路由 / 事件 -> 元件 / store -> API -> 狀態更新 / 渲染。
  - 後端請求：controller -> service -> repository / mapper -> SQL / 外部 API -> response。
  - 批次工作：trigger -> job -> service -> persistence / side effect。
- 區分證據狀態：已確認、合理推論、待驗證。
- 不要宣稱已閱讀沒有實際開啟或搜尋過的檔案。
- 不要從單一檔案、單一 controller、單一 component 或單一命名慣例推斷完整系統設計。
- 重要結論優先附上檔案與行號參考。
- 輸出長度應符合使用者請求。窄問題用精簡回答；明確要求閱讀筆記或廣泛分析時再使用結構化筆記。

## 停止條件

滿足以下任一條件即可停止擴張閱讀範圍：

- 已能直接回答使用者問題。
- 已形成足以解釋問題的最小閉環。
- 新展開的檔案只補充背景，沒有實質改變目前結論。
- 剩餘不確定性已明確落在執行時資訊、資料庫內容、環境設定、外部系統或未提供的檔案上。

## 預設忽略

除非與目前問題直接相關，否則不要優先展開：

- `node_modules`、`dist`、`build`、`.next`、`coverage`。
- lock file、編譯輸出與其他大型生成檔。
- 與目前入口無直接關聯的測試檔。
- 只做 re-export 的 barrel files。
- 純樣式檔，除非問題與 UI 呈現、class 切換或樣式條件有關。
- 與當前流程無直接關聯的第三方函式庫原始碼。

## 預設模式

根據使用者請求選擇模式。若使用者指定模式，就照指定模式執行。

- `file`：理解單一檔案的角色、結構、依賴、關鍵邏輯、風險與下一步閱讀方向。
- `feature`：追蹤使用者可見功能、端點、按鈕、路由、批次工作或跨檔流程。
- `module`：根據實際讀過的檔案，整理模組責任、主要檔案、流程、依賴、耦合與缺口。

如果使用者只提供檔案路徑，預設使用 `file`。
如果使用者提供功能名稱加入口，預設使用 `feature`。
如果使用者提供目錄或模組並要求總覽，預設使用 `module`。

## 閱讀流程

1. 用一句話確認目標：這次要理解什麼，以及為了什麼目的。
2. 整理已知輸入：`goal`、`entry`、`mode`、`max_depth`、`constraints`、`expected_output`。
3. 確認入口與模式。
4. 只在需要時做輕量專案掃描：manifest、路由表、模組目錄、框架設定或符號搜尋。
5. 先閱讀入口檔案。
6. 只追蹤相關的呼叫方、被呼叫方、import、路由、API 呼叫、SQL/mapper、DTO/entity 定義、設定或 UI 事件，並追到形成最小閉環或達到 `max_depth` 為止。
7. 閱讀時追蹤證據狀態：
   - 已確認：可從已開啟檔案或命令輸出直接看見。
   - 合理推論：根據命名、框架慣例、目錄結構或部分呼叫關係推論。
   - 待驗證：重要結論受未讀檔案、缺少執行時脈絡、隱藏 SQL、生成碼、環境設定或多種可能流程阻擋。
8. 依使用者要求的格式回答。若沒有指定格式且任務較廣，使用 `references/output-templates.md` 中對應的模板。
9. 除非回答本來就需要非常簡短，結尾應指出最高價值的下一個閱讀目標或下一步行動。

## 技術線索

優先依照專案實際檔案與慣例判斷。以下只是提醒，不是固定技術棧清單。

- Java web / Spring / Spring Boot：檢查 controller-service-repository 或 mapper 流程、DI、DTO/entity 流動、驗證、交易邊界、例外處理與設定。
- Hibernate / MyBatis / SQL-heavy 專案：檢查 DAO/mapper XML 或註解、HQL/SQL、stored procedure、result mapping、交易邊界與 entity/DTO mapping。
- Vue / React / 前端專案：檢查路由、元件邊界、props/events、store/composables/hooks、API client、狀態轉換、表單處理與渲染條件。
- jQuery / 舊式前端：檢查 DOM 綁定、初始化流程、全域狀態、AJAX 呼叫、工具函式耦合、動態 HTML 生成與表單參數組裝。
- 未知技術棧：先檢查 manifest 與入口/設定檔，再套用假設。

## 輸出指引

窄問題直接回答，並為關鍵點附上證據。

結構化筆記應包含：

- 範圍與入口。
- 實際閱讀或搜尋過的檔案。
- 已確認行為。
- 合理推論，並清楚標示。
- 待驗證項目。
- 與請求相關的依賴、呼叫方、被呼叫方或資料流。
- 相關時才列出風險或可疑區域。
- 下一步檔案或行動，以及原因。

只有在目前請求適合完整筆記格式時，才讀取 `references/output-templates.md` 的詳細模板。
