# Study Notes Markdown

當使用者希望保留教材型、教學導向、較詳細的正文內容，但又希望筆記更容易理解、閱讀、查找、回查、複習與擴充時，使用這個 skill。

這個 skill 的核心不是把內容濃縮成摘要，而是讓原本的筆記「更好用」。

---

# 適用與不適用

## 適用情境

- 課程筆記、教材筆記、技術學習筆記
- 想保留完整教學型正文，而不是做極短摘要
- 單篇筆記過長、缺乏導航、閱讀中容易迷路
- 想把零散筆記整理成可持續擴充的 Markdown 倉庫
- 想讓筆記同時支援順讀、跳讀、搜尋、回查與複習

## 不適用情境

- 使用者只要極短摘要、考前一頁速記、口訣表
- 使用者要的是文章改寫、潤稿、翻譯，而不是筆記整理
- 使用者要的是知識卡片系統，而不是教材型長文筆記
- 原文本身明顯錯亂、錯誤很多、缺乏基本邏輯，需要先重構內容本身

若使用者要的是考前速記版，應採「正文版 + 複習版」雙軌，而不是把正文直接壓縮掉。複習版規則見 [references/review-system.md](./references/review-system.md)。

# 核心目標

不要刪掉核心教學內容。  
不要把詳細說明改成只有提綱。  
不要為了整齊而犧牲理解。  
不要為了複習方便而只留下結論。  
不要讓筆記只剩好看，卻不好用。

整理後的筆記應同時更容易：

- 理解
- 閱讀
- 查找
- 回查
- 複習
- 擴充

若多個目標衝突，優先順序如下：

1. 理解
2. 表述精度
3. 回查效率
4. 結構一致性
5. 版面整齊

---

# 核心原則

## 1. 保留教材型正文

- 除非使用者明確要求重寫，否則保留原本教學型正文
- 不用摘要取代正文
- 不把段落改成只剩大綱式速記
- 允許做的是加上外層結構、導航、標示與少量輔助

## 2. 先補結構，再碰內容

整理順序優先是：

1. 倉庫入口與章節導航
2. 判斷是否需要拆分大筆記
3. 單篇筆記開頭與回查入口
4. 高價值內部連結、關鍵字、索引或 FAQ
5. 複習支架與自測入口
6. 最後才是小幅內容微調

## 3. 小改穩改，不過度濃縮

- 優先做可重複套用的結構化修改
- 不做大量語意重寫
- 不追求把每篇都整理成考前速記
- 不追求每段都加總結，避免破壞原文節奏

## 4. 理解優先於整齊

若形式上的整齊會讓內容更難懂，應保留原本較易理解的寫法。

## 5. 查找與複習不是附屬功能

- 好的筆記不只要能讀，還要能找
- 關鍵字、別名、英文術語、常見問法、FAQ、索引，都是查找入口
- 複習支架應幫助提取，不應製造第二份互相矛盾的內容

---

# 工作流程

1. 檢查整體倉庫結構與章節分布
2. 判斷目前是整份倉庫、章節整理，還是單篇筆記整理
3. 判斷當前筆記是大筆記還是小筆記
4. 若為大筆記，先拆分，再整理拆分後的小筆記
5. 先補根目錄 `README.md` / 章節 `README.md` / 單篇入口
6. 視需要補關鍵字系統、內部連結、FAQ、術語表、索引
7. 視需要補 30 秒複習入口、速查區、自測問題、章節複習路線
8. 最後再做少量內容微調
9. 檢查是否損失表述精度、是否產生壞鏈、是否真的提升理解與回查效率

---

# 執行硬規則

- 若已判定為大筆記，不要只補文首導讀就結束；應先完成拆分，再整理拆分後的檔案
- 若同時符合大筆記條件與小筆記條件，優先按大筆記處理
- 若最終決定不拆，必須能說明不拆的理由，例如篇幅可控、主題高度集中、拆分反而破壞理解
- 若拆分後會產生一組子筆記，應建立對應入口頁 `README.md`
- 若整理後改動了標題、檔名、閱讀順序或入口連結，必須同步檢查章節與根目錄 `README.md`
- 若同義詞、英文術語或常見問法明顯存在，不應省略關鍵字系統
- 若主題具有高頻回查價值，不應只補標題而略過速查區或回查入口

# 大筆記與小筆記的判斷

## 視為大筆記的情況

符合以下任一情況時，應視為大筆記並先拆分：

- 單篇過長，閱讀與搜尋成本高
- 同時涵蓋多個明顯子主題
- 已有穩定的標題階層，例如 `#`、`##`、`###`
- 單篇已同時承擔總覽、定義、比較、實作、例子等多種任務
- 使用者在單篇中不容易快速定位內容

## 建議的拆分門檻

符合以下任一門檻時，預設應拆分，除非有明確理由證明不拆更好：

- 單篇超過 `1500` 行
- 單篇有 `5` 個以上一級主題，或 `10` 個以上二級主題
- 同一篇同時包含概念講解與多個獨立案例、題解或實作主題
- 單篇已接近一本小冊，讀者難以靠搜尋與標題快速定位

## 視為小筆記的情況

符合以下情況時，可直接整理：

- 單篇主題明確且範圍集中
- 長度可控
- 適合獨立閱讀與獨立回查
- 不拆分也不影響理解與維護

## 拆分實作規則

- 優先沿用原本穩定的標題階層拆分，不要為拆而拆
- 優先拆成「主題資料夾 + README + 多篇小筆記」的形式
- 拆分後的小筆記應能獨立閱讀，並保有必要上下文
- 拆分後應補上前後閱讀順序與返回入口連結，避免讀者迷路
- 檔名、資料夾名與連結規則見 [references/style-rules.md](./references/style-rules.md)

# 倉庫層級設計

整份筆記倉庫至少應有三層入口：

1. 根目錄入口
2. 章節入口
3. 單篇入口

若缺任一層，閱讀與回查效率通常會明顯下降。

---

# README 同步規則

整理單篇筆記時，應檢查該筆記所在章節的 `README.md`：

- 若章節 `README.md` 不存在，且單篇筆記已連到 `./README.md`，應建立章節 `README.md`
- 若章節 `README.md` 已存在，但未包含本篇筆記，應同步補上本篇連結
- 若整理後改動了標題、檔名、閱讀順序、延伸閱讀或回查入口，應同步更新章節 `README.md`
- 若章節 `README.md` 在 Git 狀態中顯示為刪除，應明確說明；若整理後會造成壞鏈，應重建或請使用者確認

整理單篇筆記時，也應檢查根目錄 `README.md` 的目前進度、章節導航與快速入口是否仍準確。只要符合以下情況，就必須同步：

- 新增、刪除、移動章節資料夾
- 新增章節 `README.md`
- 章節導航或整體學習路徑發生變化
- 根目錄 README 的章節入口缺失或已失效
- 根目錄 README 的目前進度與已整理小節不一致
- 單篇整理後新增了 FAQ、索引頁、術語表或高頻速查入口，導致總目錄應補入口

---

# 單篇筆記的最低必要輸出

每篇整理後的單篇筆記，至少應具備：

1. 清楚的標題
2. 所屬章節或返回章節入口
3. 本節導讀
4. 關鍵字
5. 建議回查情境
6. 正文教學內容

符合以下任一情況時，應再補：

- 主題有高頻回查需求
- 有規則、公式、判準、條件、例外
- 容易混淆
- 使用者明確提到複習、自測、考前回看、面試或速查

此時應補：

- `30 秒複習入口`
- `速查區`
- 視需要補 `自測問題`

完整欄位與模板見 [references/templates.md](./references/templates.md)。

# 何時讀取 references

為了控制上下文，先做核心判斷，再按需讀取以下檔案：

- [references/templates.md](./references/templates.md)
  - 要整理根目錄、章節 README、單篇筆記，或需要輸出骨架時讀
- [references/search-system.md](./references/search-system.md)
  - 主題有同義詞、英文術語、常見問法、FAQ、索引、術語表需求時讀
- [references/review-system.md](./references/review-system.md)
  - 使用者提到複習、自測、考前回看、面試速答，或主題屬於高頻回查頁時讀
- [references/style-rules.md](./references/style-rules.md)
  - 要拆分大筆記、調整檔名、資料夾名、標題、內容標記或內部連結時讀
- [references/quality-checklist.md](./references/quality-checklist.md)
  - 大量整理、多檔改動、準備收尾驗收，或懷疑模板化過度時讀

不要一次載入所有 references；只讀當前步驟需要的部分。

# 最終準則

這個 skill 的終極目標，不是讓筆記變得更像模板。  
而是讓筆記在保留知識密度與教學價值的前提下，變成真正可理解、可閱讀、可查找、可複習、可擴充的長期資產。
