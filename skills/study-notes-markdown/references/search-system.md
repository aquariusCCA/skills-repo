# 查找系統設計

當筆記存在別名、英文術語、常見問法、FAQ、索引或術語表需求時，讀取本檔。

---

## 目標

好筆記不只要能讀，還要能找。  
查找能力的重點是降低「我知道內容在這裡，但一時找不到」的成本。

---

## 最低要求

至少提供以下四種入口中的兩種；若為大型倉庫，建議至少三種：

1. 關鍵字欄位
2. README 導航
3. 索引 / FAQ / 術語表
4. 內部連結與前後篇連結

---

## 關鍵字規則

- 關鍵字必須服務搜尋，而不是只服務美觀
- 同一概念若有中英文、縮寫、別名，應盡量補齊
- 若常見搜尋詞與正式術語不同，應同時保留
- 若存在高混淆詞，應在關鍵字或易混淆區中明確標示

建議格式：

- 主題：xxx
- 別名：xxx、xxx
- 英文：xxx
- 常見搜尋：xxx、xxx
- 易混淆：xxx vs xxx

---

## 何時建立 FAQ / 索引 / 術語表

符合以下情況時，應考慮建立：

- 同一術語在多篇出現
- 使用者常用問句而不是術語來查找
- 同義詞很多，搜尋命中不穩
- 章節內容多，僅靠 README 很難快速定位
- 存在高頻回查主題，例如公式、判準、差異比較、流程

---

## 常見查詢詞前置原則

若未來使用者很可能會用以下方式搜尋，應在筆記中前置這些詞：

- 問句式，例如「什麼是...」「差別是什麼」「什麼時候用...」
- 縮寫式，例如 API、MVC、ACID、CAP
- 中英夾雜式，例如 索引 index、事務 transaction
- 題型式，例如 判斷條件、流程、公式、例外、比較

---

## 各層級建議

### 單篇筆記

- 補齊結構化關鍵字
- 在導讀或速查區前置常見問法
- 補前後篇與相關篇連結

### 章節 README

- 標示本章高頻回查頁
- 標示易混淆主題
- 說明先看哪篇比較不會卡

### 根目錄 README

- 提供章節導航
- 提供高頻查找主題入口
- 必要時補 FAQ / 索引 / 術語表入口

---

## 避免的做法

- 只寫 2 到 3 個模糊關鍵字
- 只有正式術語，沒有常見搜尋詞
- 將 FAQ 寫成另一份重複正文
- 有入口名稱但沒有實際鏈接
