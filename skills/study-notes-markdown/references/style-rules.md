# 命名、連結與內容標記規則

當需要拆分大筆記、調整檔名、資料夾名、標題、內部連結或標記欄位時，讀取本檔。

---

## 檔名規則

- 拆分後的檔名應具可回查性，例如 `01 概述.md`、`02 單向鏈表.md`
- 若原檔名已帶有明確序號，拆分後應保留該序號，避免破壞閱讀順序
- 若原檔名過於泛化，例如 `README.md`、`筆記.md`、`總結.md`，可改用更具主題性的資料夾名
- 若同名資料夾已存在，先判斷是否本來就是這篇筆記的拆分目錄；若不是，應改用更清楚且不衝突的名稱

---

## 拆分資料夾規則

- 拆分資料夾預設使用原筆記檔名去掉 `.md` 後的名稱
- 優先拆成「主題資料夾 + README + 多篇小筆記」
- 若原筆記已有章節層級的 `images/` 目錄，拆分後的小筆記預設沿用原圖片，不複製資產
- 若原文中存在概念篇、例題篇、實作篇，優先分開而不是混在同一檔

---

## 標題規則

- 標題應清楚反映主題，不用過度文學化
- 優先讓未來的自己能看懂，而不是追求短
- 若為題解或案例，標題應含題號、名稱或問題類型
- 若整個倉庫已有慣用語言或字形，應延續既有慣例，不要局部改風格

---

## 連結規則

- 章節與根目錄入口應形成雙向或可返回的導航
- 單篇拆分後應補上前後閱讀順序與返回入口連結
- 若調整檔名、資料夾名、標題，應同步更新所有受影響的內部連結
- 不要留下只提名稱卻沒有實際鏈接的入口

---

## README 同步規則

整理單篇筆記時，應檢查該筆記所在章節的 `README.md`：

- 若章節 `README.md` 不存在，且單篇筆記已連到 `./README.md`，應建立章節 `README.md`
- 若章節 `README.md` 已存在，但未包含本篇筆記，應同步補上本篇連結
- 若整理後改動了標題、檔名、閱讀順序、延伸閱讀或回查入口，應同步更新章節 `README.md`
- 若章節 `README.md` 在 Git 狀態中顯示為刪除，應明確說明；若整理後會造成壞鏈，應重建或請使用者確認

整理單篇筆記時，也應檢查根目錄 `README.md` 的目前進度、章節導航與快速入口是否仍準確。只要符合以下情況，就必須同步：

- 新增、刪除、移動章節資料夾
- 新增章節 `README.md`
- 章節導航或整體學習路徑發生變化
- 根目錄 README 的章節入口缺失或已失效
- 根目錄 README 的目前進度與已整理小節不一致
- 單篇整理後新增了 FAQ、索引頁、術語表或高頻速查入口，導致總目錄應補入口

若目前看不到根目錄或章節入口檔，應在收尾報告中列為「未驗證同步項目」。

---

## 建議標記類型

視主題需要，可在單篇筆記中穩定使用以下欄位：

- `本節導讀`
- `關鍵字`
- `建議回查情境`
- `30 秒複習入口`
- `速查區`
- `自測問題`
- `延伸閱讀`

速查區常用子欄位：

- `核心定義`
- `關鍵規則 / 公式 / 判準`
- `前提條件`
- `例外情況`
- `常見錯誤`
- `一句話對比`

---

## 標記使用原則

- 模板可以套用，但不能機械式硬套
- 若欄位會破壞閱讀流暢度，可調整或省略局部欄位
- 不應為了欄位完整而改壞原文節奏
- 同一區域不要同時塞入兩份功能重疊的資訊
