# Repair Study Notes

使用這個 skill 的目標，不是把筆記排整齊，而是把原本難學、難懂、可能有錯的內容，重構成真正能拿來學習、複習與回查的版本。

---

## 參考檔使用時機

主檔只保留每次幾乎都需要的核心規則。需要細節時再讀對應 reference：

- 需要看常見內容問題與對應處理策略時，讀 [references/problem-patterns.md](references/problem-patterns.md)
- 需要可直接套用的輸出骨架、修正說明、待查證與衝突處理格式時，讀 [references/templates.md](references/templates.md)
- 需要拆分規則、fallback 細則、術語統一檢查、證據門檻與最終驗收清單時，讀 [references/quality-controls.md](references/quality-controls.md)

不要預設把三份 reference 一次全部載入。

## 這個 skill 真正在處理什麼

這個 skill 處理的是「內容層的修復」，不是「外層結構的美化」。

### repair-study-notes 處理的事

- 釐清主線
- 修正錯誤知識或錯誤推論
- 消解上下文矛盾
- 統一術語
- 重排敘述順序
- 讓例子、規則、結論互相對上
- 把逐字稿感內容重構成可學習的筆記

### repair-study-notes 不優先處理的事

- 補標題
- 補導讀
- 補導航
- 補 README
- 整份倉庫的章節入口
- 單純排版整理
- 只補關鍵字與回查索引

簡化判斷：

```text
repair-study-notes = 修內容
study-notes-markdown = 整入口、導航與倉庫結構
```

## 何時使用

當使用者要求修復、重構、改寫或校正某個學習筆記檔案，且符合以下任一情況時，使用本 skill：

- 原文表達難懂，讀者不容易抓到主線
- 同一篇內前後說法矛盾
- 例子、結論、定義彼此對不上
- 知識點順序混亂，導致理解成本高
- 內容摻雜錯誤知識、錯誤推論或錯誤術語
- 原文像口語逐字稿、想到哪寫到哪，難以作為學習筆記
- 原文雖然資訊很多，但不適合直接複習與回查
- 程式碼與文字解說互相衝突
- 原文重複過多，主線被噪音淹沒

## 何時不要使用

若原文內容本身清楚，只是缺少以下外層結構，則不要優先使用本 skill，而應改用一般整理型 skill，例如 `study-notes-markdown`：

- 缺標題
- 缺導讀
- 缺導航
- 缺 README
- 排版凌亂但內容本身正確且可讀
- 只需要補關鍵字、章節入口、回查索引

## 規則優先級

若不同規則彼此衝突，依以下順序處理：

1. 使用者當前明確要求
2. 不得偽造確定性、不得掩蓋待查證項目
3. 本 skill 的內容修復流程與證據門檻
4. 預設輸出契約與模板建議

補充：

- 使用者要求覆寫原檔、另存新檔、只回報摘要或只要重構後正文時，可以調整交付形式
- 但即使輸出變短，也不可省略真正存在的衝突、錯誤與待查證資訊
- 模板是預設骨架，不是高於使用者要求的硬性格式

## 輸入與輸出範圍契約

### 1. 預設處理單位

- 預設以「單篇筆記」為主要處理單位
- 若使用者一次交付多篇內容，可逐篇處理；不要把多篇不同主題硬合成一篇
- 若原文其實是同一主題的長篇內容，可先判斷是否應拆分成多篇

### 2. 檔案與內容型態

本 skill 主要適用於：

- Markdown 學習筆記
- 課程筆記
- 逐字稿式整理稿
- 技術概念筆記
- 法規、制度、流程型筆記
- 含少量到中量程式碼的教學筆記

若原文是高度表格化、公式密集、圖表密集、或大量程式碼主導的內容，仍可使用本 skill，但必須更保守，不得憑模糊印象補全缺失資訊。

### 3. 交付模式

- 預設交付為「重構後版本 + 簡短交付說明」
- 若使用者明確要求固定格式收尾，可採完整報告式輸出
- 若使用者要求直接覆寫原檔、極短回覆或只要正文，可壓縮說明，但不可隱藏修正決策與待查證項目

### 4. 覆寫與另存

- 若使用者明確要求覆寫原檔，才採覆寫策略
- 若使用者未明講，預設以「產出整理後版本」為主，不自行假設已覆寫成功
- 若內容改動幅度很大，較適合另存新檔，避免覆蓋原始材料

### 5. 極短內容

即使原文很短，只要存在實質內容問題，仍適用本 skill。  
但若內容極短且問題單純，可在保持輸出契約的前提下簡化篇幅，不必為了格式而冗長。

## 核心目標

整理完成後，筆記應盡量達成以下效果：

1. 讀者能快速知道這篇到底在講什麼。
2. 讀者能分清楚定義、條件、推論、例子與例外。
3. 原文中的矛盾、錯誤與跳躍敘述已被修正或明確標示。
4. 全文術語一致，內容順序更符合學習路徑。
5. 內容既能順讀，也能回查，後續擴充不會越改越亂。

## 核心原則

### 1. 忠於原意，但不忠於原句

- 保留原本想表達的知識意圖
- 不必死守原句、原段順序與原本口語表達
- 允許把口語改成教學型表達
- 允許把一大段拆成多個清楚小節
- 允許把分散在不同位置的同類內容合併

### 2. 理解優先於形式保留

- 若原文形式妨礙理解，可改寫
- 若原文順序讓讀者難懂，可重排
- 若原文有重複、繞圈、插話，可刪減噪音
- 若原文像逐字稿，應轉成真正可讀的筆記

### 3. 先修內容，再補外層結構

處理順序固定如下：

1. 看懂原文在講什麼
2. 找出矛盾、錯誤、跳步、混淆點
3. 抽出知識骨架
4. 依骨架重寫內容
5. 最後才補導讀、關鍵字、回查入口與延伸閱讀

### 4. 錯誤不能直接沿用

若原文存在明顯錯誤，不得只是照抄整理，必須採取以下其中一種做法：

1. 直接修正為較可靠版本
2. 標示原文有誤，並給出修正版
3. 若無法確認正誤，明確標示為「待查證」

### 5. 矛盾必須顯性處理

若同一篇筆記出現前後矛盾：

- 不可只保留其中一段而不說明
- 不可把互相衝突的內容並列卻不處理
- 能判斷時，採用較可靠版本並統一全文
- 不能判斷時，明確標示衝突點與待確認處

### 6. 重構不是過度濃縮

- 不把完整教學內容縮成只有提綱
- 不把推理刪成只剩結論
- 不為了短而犧牲理解

### 7. 不得用猜測補成確定知識

- 若原文沒有提供足夠條件，不得自行補出關鍵定義、例子、公式、例外規則或結論
- 若只能憑模糊常識或零散印象推測，不得直接寫成肯定句
- 可以整理成更穩定的教學骨架，但不得假裝原文明確說過沒有說過的事

## 修正知識時的證據門檻

只有在原文上下文、例子、流程、公式、程式碼或同篇較明確定義提供足夠支持時，才可直接改成較可靠版本。

若缺乏支持，不得因「看起來怪」就硬改，應列入「待查證 / 不確定處」。

法規、金融制度、醫學與健康、精確數值或版本差異明顯的內容，要更保守。  
需要逐項檢查時，讀 [references/quality-controls.md](references/quality-controls.md)。

---

## 判斷優先級

當原文出現衝突、混用或不確定時，依以下順序判斷哪個版本較可靠：

1. 明確定義是否一致
2. 例子是否支持該說法
3. 程式碼、流程或公式是否支持該說法
4. 上下文前後是否一致
5. 是否符合一般穩定知識與術語習慣

若前 3 項無法支持任何一方，不要硬選，改列為「待查證」。

---

## 程式碼與文字衝突處理

若原文中的程式碼與文字解說互相衝突，優先採用較可驗證的一方；若兩者都不足以證成結論，就保留衝突並列入待查證。

需要具體判斷步驟時，讀 [references/problem-patterns.md](references/problem-patterns.md) 與 [references/quality-controls.md](references/quality-controls.md)。

---

## 超長內容處理優先順序

長文優先穩住主題、核心定義、明顯錯誤與衝突，再處理術語、噪音與輔助結構。

不要在主線尚未穩定前，把篇幅優先花在潤句或外層裝飾。  
需要完整優先順序時，讀 [references/quality-controls.md](references/quality-controls.md)。

## 強制工作流程

每次使用本 skill，必須依照以下流程處理，不可跳步。

### 第一步：判斷是否真的需要內容重構

符合以下任一情況，直接視為需要重構：

- 原文不是只缺結構，而是本身就難懂
- 讀完後無法清楚說出核心結論
- 同篇內出現明顯矛盾
- 有錯誤知識
- 原文像逐字稿，不適合直接作為筆記
- 例子和規則對不上
- 程式碼與文字解釋互相衝突

若只是缺標題、導讀、導航、README 或純排版問題，則不要用本 skill 當主要 skill。

### 第二步：抽出原文知識骨架

在正式改寫前，先辨識：

1. 這篇的主題是什麼
2. 作者真正想解決的問題是什麼
3. 核心定義有哪些
4. 條件與結論有哪些
5. 例子是在支持哪個規則
6. 哪些地方互相矛盾
7. 哪些地方可能有錯
8. 哪些地方其實資訊不足，不能硬補

若骨架尚未清楚，不要急著潤飾句子。

### 第三步：決定重構幅度

依內容狀況，只能選擇以下三種之一，並讓最終輸出與之相符：

- 輕度修復：原文本身大致正確，幾乎沒有矛盾，只有少量句子難懂
- 中度改寫：主體其實正確，只是表達繞、順序不好或術語略亂
- 大幅重寫：原文像逐字稿、前後矛盾、錯知識多、重複多，難以直接作為教材或複習筆記

### 第四步：判斷是否拆分

若內容很長，且同時混有多個子主題，先判斷是否應拆成多篇：

- 若拆分後明顯更利於回查與學習，優先拆分
- 若主線其實仍集中在同一核心問題，可不拆，但需控制結構層次
- 拆分後要交代閱讀順序與各篇用途
- 若本次未拆分，也可在修正決策中說明原因

### 第五步：建立新結構

除非內容極短，否則優先使用以下骨架：

1. 這篇在解決什麼問題
2. 先講結論 / 核心理解
3. 核心定義 / 規則
4. 判斷條件或推理過程
5. 例子說明
6. 常見混淆點 / 易錯點
7. 修正說明 / 待查證（如有）
8. 一句話抓核心

需要完整骨架與格式範例時，讀 [references/templates.md](references/templates.md)。

### 第六步：重寫內容

改寫時必須做到：

- 句子更短
- 指代更清楚
- 條件更明確
- 概念先後更合理
- 同一術語全篇一致
- 結論與例子能互相對上
- 對錯誤與衝突採取顯性處理
- 對不確定處保留待查證，而不是硬改

若遇到具體問題類型，例如原文難懂、前後矛盾、順序失衡、重複噪音，讀 [references/problem-patterns.md](references/problem-patterns.md)。

### 第七步：補必要的輔助結構

只有在內容骨架已穩定後，才補：

- 本節導讀
- 關鍵字
- 建議回查情境
- 延伸閱讀
- 常見回查問題

### 第八步：交付前做收尾檢查

若需要拆分、統一術語、處理 fallback、檢查證據門檻或逐項驗收，讀 [references/quality-controls.md](references/quality-controls.md)。

## 預設最終輸出契約

只要使用本 skill，交付結果都必須讓讀者看得出以下四類資訊：

1. 重構後內容是什麼
2. 原文主要問題是什麼
3. 這次做了哪些修正決策
4. 是否仍有待查證 / 不確定處

預設情況下，建議用以下四部分顯式輸出。
若使用者要求覆寫原檔、簡短回報或只要正文，可把 B、C、D 壓縮成簡短交付說明或內嵌註記；但不可因格式縮短而省略這些資訊。

### A. 重構後正文

給出整理後、可直接閱讀與複習的版本。

### B. 原文主要問題摘要

用簡短條列說明原文主要問題，至少涵蓋實際出現者：

- 難懂
- 前後矛盾
- 錯誤知識
- 術語混用
- 順序失衡
- 重複噪音
- 程式碼與文字衝突
- 資訊不足但原文講得像很確定

### C. 本次修正決策

至少交代：

- 採用的是輕度修復 / 中度改寫 / 大幅重寫
- 有無統一術語
- 有無刪除重複噪音
- 有無處理衝突
- 有無保留待查證處
- 有無拆分，或為何未拆分

### D. 待查證 / 不確定處

若存在無法確認的部分，必須明確列出。  
若沒有，寫「本次未保留待查證項目」。

若使用者明確要求只交付正文，至少要把待查證與重大修正決策內嵌在正文或收尾一句話中，避免資訊遺失。  
需要具體輸出模板與修正/衝突格式時，讀 [references/templates.md](references/templates.md)。

## 成功標準

完成後，應盡量滿足以下可檢查條件：

1. 能用一到兩句話說清楚這篇在解決什麼問題。
2. 正文中能明確辨識核心定義、條件、結論與例子。
3. 已發現的衝突沒有被直接並列放過，而是已修正或被標示。
4. 全文主要術語一致，未出現未交代的混用。
5. 錯誤知識已修正，或被明確列入待查證。
6. 例子、規則、結論彼此能對上。
7. 輸出符合最終輸出契約。
8. 內容既能順讀，也能回查。

---

## 禁止事項

- 不可把錯誤知識包裝得更流暢卻不修正
- 不可只調排版、不處理內容矛盾
- 不可將互相衝突的結論同時保留為正確
- 不可過度濃縮成只有考前速記
- 不可為了保留原味而犧牲可理解性
- 不可在無把握時假裝確定
- 不可省略原文問題摘要與修正決策
- 不可把待查證內容偽裝成已確認知識
- 不可憑模糊印象補出原文沒有提供的關鍵條件、例子、例外與結論

這個 skill 的任務不是「把筆記排整齊」，而是把原本難學、難懂、可能有錯的筆記，重構成真正能拿來學習的版本。
