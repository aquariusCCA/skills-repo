# Repair Study Notes

使用這個 skill 的目標，不是把筆記排整齊，而是把原本難學、難懂、可能有錯的內容，重構成真正能拿來學習、複習與回查的版本。

## 參考檔使用時機

主檔只保留每次幾乎都需要的核心規則。需要細節時再讀對應 reference：

- 需要看常見內容問題與對應處理策略時，讀 [references/problem-patterns.md](references/problem-patterns.md)
- 需要可直接套用的輸出骨架、修正說明、待查證與衝突處理格式時，讀 [references/templates.md](references/templates.md)
- 需要拆分規則、fallback 細則、術語統一檢查與最終驗收清單時，讀 [references/quality-controls.md](references/quality-controls.md)

不要預設把三份 reference 一次全部載入。

---

## 何時使用

當使用者要求修復、重構、改寫或校正某個學習筆記檔案，且符合以下任一情況時，使用本 skill：

- 原文表達難懂，讀者不容易抓到主線
- 同一篇內前後說法矛盾
- 例子、結論、定義彼此對不上
- 知識點順序混亂，導致理解成本高
- 內容摻雜錯誤知識、錯誤推論或錯誤術語
- 原文像口語逐字稿、想到哪寫到哪，難以作為學習筆記
- 原文雖然資訊很多，但不適合直接複習與回查
- 程式碼與文字解說互相衝突
- 原文重複過多，主線被噪音淹沒

## 何時不要使用

若原文內容本身清楚，只是缺少以下外層結構，則不要優先使用本 skill，而應改用一般整理型 skill，例如 `study-notes-markdown`：

- 缺標題
- 缺導讀
- 缺導航
- 缺 README
- 排版凌亂但內容本身正確且可讀
- 只需要補關鍵字、章節入口、回查索引

簡化判斷：

```text
repair-study-notes = 修內容
study-notes-markdown = 整入口、導航與倉庫結構
```

## 核心目標

整理完成後，筆記應盡量達成以下效果：

1. 讀者能快速知道這篇到底在講什麼。
2. 讀者能分清楚定義、條件、推論、例子與例外。
3. 原文中的矛盾、錯誤與跳躍敘述已被修正或明確標示。
4. 全文術語一致，內容順序更符合學習路徑。
5. 內容既能順讀，也能回查，後續擴充不會越改越亂。

## 核心原則

### 1. 忠於原意，但不忠於原句

- 保留原本想表達的知識意圖
- 不必死守原句、原段順序與原本口語表達
- 允許把口語改成教學型表達
- 允許把一大段拆成多個清楚小節
- 允許把分散在不同位置的同類內容合併

### 2. 理解優先於形式保留

- 若原文形式妨礙理解，可改寫
- 若原文順序讓讀者難懂，可重排
- 若原文有重複、繞圈、插話，可刪減噪音
- 若原文像逐字稿，應轉成真正可讀的筆記

### 3. 先修內容，再補外層結構

處理順序固定如下：

1. 看懂原文在講什麼
2. 找出矛盾、錯誤、跳步、混淆點
3. 抽出知識骨架
4. 依骨架重寫內容
5. 最後才補導讀、關鍵字、回查入口與延伸閱讀

### 4. 錯誤不能直接沿用

若原文存在明顯錯誤，不得只是照抄整理，必須採取以下其中一種做法：

1. 直接修正為正確版本
2. 標示原文有誤，並給出修正版
3. 若無法確認正誤，明確標示為「待查證」

### 5. 矛盾必須顯性處理

若同一篇筆記出現前後矛盾：

- 不可只保留其中一段而不說明
- 不可把互相衝突的內容並列卻不處理
- 能判斷時，採用較可靠版本並統一全文
- 不能判斷時，明確標示衝突點與待確認處

### 6. 重構不是過度濃縮

- 不把完整教學內容縮成只有提綱
- 不把推理刪成只剩結論
- 不為了短而犧牲理解

## 判斷優先級

當原文出現衝突、混用或不確定時，依以下順序判斷哪個版本較可靠：

1. 明確定義是否一致
2. 例子是否支持該說法
3. 程式碼、流程或公式是否支持該說法
4. 上下文前後是否一致
5. 是否符合一般穩定知識與術語習慣

若前 3 項無法支持任何一方，不要硬選，改列為「待查證」。

## 強制工作流程

每次使用本 skill，必須依照以下流程處理，不可跳步。

### 第一步：判斷是否真的需要內容重構

符合以下任一情況，直接視為需要重構：

- 原文不是只缺結構，而是本身就難懂
- 讀完後無法清楚說出核心結論
- 同篇內出現明顯矛盾
- 有錯誤知識
- 原文像逐字稿，不適合直接作為筆記
- 例子和規則對不上
- 程式碼與文字解釋互相衝突

若只是缺標題、導讀、導航、README 或純排版問題，則不要用本 skill 當主要 skill。

### 第二步：抽出原文知識骨架

在正式改寫前，先辨識：

1. 這篇的主題是什麼
2. 作者真正想解決的問題是什麼
3. 核心定義有哪些
4. 條件與結論有哪些
5. 例子是在支持哪個規則
6. 哪些地方互相矛盾
7. 哪些地方可能有錯

若骨架尚未清楚，不要急著潤飾句子。

### 第三步：決定重構幅度

依內容狀況，只能選擇以下三種之一，並讓最終輸出與之相符：

- 輕度修復：原文本身大致正確，幾乎沒有矛盾，只有少量句子難懂
- 中度改寫：主體其實正確，只是表達繞、順序不好或術語略亂
- 大幅重寫：原文像逐字稿、前後矛盾、錯知識多、重複多，難以直接作為教材或複習筆記

### 第四步：建立新結構

除非內容極短，否則優先使用以下骨架：

1. 這篇在解決什麼問題
2. 先講結論 / 核心理解
3. 核心定義 / 規則
4. 判斷條件或推理過程
5. 例子說明
6. 常見混淆點 / 易錯點
7. 修正說明 / 待查證（如有）
8. 一句話抓核心

需要完整骨架與格式範例時，讀 [references/templates.md](references/templates.md)。

### 第五步：重寫內容

改寫時必須做到：

- 句子更短
- 指代更清楚
- 條件更明確
- 概念先後更合理
- 同一術語全篇一致
- 結論與例子能互相對上

若遇到具體問題類型，例如原文難懂、前後矛盾、順序失衡、重複噪音，讀 [references/problem-patterns.md](references/problem-patterns.md)。

### 第六步：補必要的輔助結構

只有在內容骨架已穩定後，才補：

- 本節導讀
- 關鍵字
- 建議回查情境
- 延伸閱讀
- 常見回查問題

### 第七步：交付前做收尾檢查

若需要拆分、統一術語、處理 fallback 或逐項驗收，讀 [references/quality-controls.md](references/quality-controls.md)。

## 強制最終輸出契約

只要使用本 skill，最終輸出至少必須包含以下四部分：

### A. 重構後正文

給出整理後、可直接閱讀與複習的版本。

### B. 原文主要問題摘要

用簡短條列說明原文主要問題，至少涵蓋實際出現者：

- 難懂
- 前後矛盾
- 錯誤知識
- 術語混用
- 順序失衡
- 重複噪音
- 程式碼與文字衝突

### C. 本次修正決策

至少交代：

- 採用的是輕度修復 / 中度改寫 / 大幅重寫
- 有無統一術語
- 有無刪除重複噪音
- 有無處理衝突
- 有無保留待查證處

### D. 待查證 / 不確定處

若存在無法確認的部分，必須明確列出。
若沒有，寫「本次未保留待查證項目」。

不得只輸出重構正文而省略 B、C、D。

需要具體輸出模板與修正/衝突格式時，讀 [references/templates.md](references/templates.md)。

## 成功標準

完成後，應盡量滿足：

1. 讀者可以用自己的話複述核心概念。
2. 讀者能說出條件、結論與例子的關係。
3. 全文術語一致，前後沒有未處理的矛盾。
4. 錯誤知識已修正或被明確標示。
5. 內容既能順讀，也能回查。
6. 最終輸出符合本 skill 的輸出契約。

---

## 禁止事項

- 不可把錯誤知識包裝得更流暢卻不修正
- 不可只調排版、不處理內容矛盾
- 不可將互相衝突的結論同時保留為正確
- 不可過度濃縮成只有考前速記
- 不可為了保留原味而犧牲可理解性
- 不可在無把握時假裝確定
- 不可省略原文問題摘要與修正決策
- 不可把待查證內容偽裝成已確認知識
