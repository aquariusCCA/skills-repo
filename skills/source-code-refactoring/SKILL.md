# 原始碼重構

使用這個 skill 來安全重構既有程式碼。目標是改善結構、命名、邊界或重複度，同時保留既有行為，除非使用者明確允許行為變更。

這是修改型 skill。不要停在分析或建議；若使用者要求重構，應在建立足夠上下文後直接修改、驗證並回報結果。

## 建議輸入

能提供越多越好，但不必每次都完整提供。

- `goal`：這次重構要達成什麼，例如抽取函式、拆分模組、整理命名、降低重複、降低耦合。
- `entry`：入口檔案、類別、方法、模組、功能、路由、端點或 UI 動作。
- `reading_note`：可選。由 `source-code-reading` 產出的 Markdown 筆記路徑。
- `refactor_type`：可選。`rename`、`extract`、`split-module`、`deduplicate`、`dependency-cleanup`、`dead-code-removal`、`api-preserving-redesign`。
- `behavior_change_allowed`：可選，預設 `false`。除非使用者明確允許，不要改變外部可觀察行為。
- `test_scope`：可選。指定要跑的測試、build、lint 或手動驗證範圍。
- `constraints`：可選。限制修改範圍、不要碰的檔案、技術棧、時間、輸出格式或相容性要求。

## 核心規則

- 預設保留行為。若需要行為變更才能完成目標，先明確指出並取得使用者同意。
- 先理解再改動。陌生程式碼、跨檔流程或不清楚的入口，先使用 `source-code-reading` 的方法建立最小閉環。
- 若提供 `reading_note`，先讀筆記，但仍要重新開啟與本次重構直接相關的檔案確認現況。
- 只把 `reading_note` 中的「已確認行為」當成保留行為的依據；對「合理推論」要回到程式碼找證據；對「待驗證」不要順手解決，除非使用者列為目標。
- 控制變更範圍。避免把格式化、命名、架構調整、測試改寫與行為修正混在同一個不必要的大改動裡。
- 不要重寫整個模組來完成局部重構，除非使用者的目標就是大範圍重寫。
- 不要移除看似未使用但可能由反射、框架、設定、模板、字串路由、外部 API 或資料庫引用的程式碼，除非已搜尋並驗證。
- 保留公開 API、資料格式、錯誤處理、權限、交易邊界、排序、預設值、空值處理與副作用，除非使用者明確要求變更。
- 修改前注意工作區可能已有使用者變更。不要還原或覆蓋無關變更；若相關檔案已有變更，讀懂後在其上工作。
- 驗證應和風險相稱。窄重構跑目標測試；跨模組重構至少跑相關測試、build 或 typecheck；無法執行時要說明原因。

## 使用 reading_note 的流程

如果使用者提供 `reading_note`：

1. 先閱讀筆記，萃取重構目標、入口、已閱讀檔案、已確認行為、合理推論、待驗證項目、風險與下一步建議。
2. 重新開啟本次重構直接相關的檔案，確認筆記沒有過期。
3. 將已確認行為轉成保護條件：測試、斷言、手動檢查點或重構中不可破壞的規則。
4. 對合理推論找程式碼證據；找不到時標成風險，不要把它當作重構依據。
5. 對待驗證項目決定處理方式：若會阻擋重構就先驗證；若不阻擋，保留在回報中。

## 重構流程

1. 用一句話確認重構目標與是否允許行為變更。
2. 整理已知輸入：`goal`、`entry`、`reading_note`、`refactor_type`、`behavior_change_allowed`、`test_scope`、`constraints`。
3. 建立上下文：
   - 若有 `reading_note`，先讀筆記再抽查相關檔案。
   - 若沒有 `reading_note` 且程式碼不熟，使用 `source-code-reading` 的最小閉環方法閱讀相關流程。
4. 找出行為保護點：現有測試、可新增的窄測試、型別檢查、build、lint、手動檢查或呼叫鏈不變式。
5. 選擇最小可完成目標的重構策略。若任務較複雜，先分成小步驟再修改。
6. 修改程式碼。每一步都保持可解釋、可回退、可驗證。
7. 更新或新增測試。只在能有效保護重構目標時新增測試，不要為了形式增加低價值測試。
8. 跑目標驗證命令。若驗證失敗，判斷是重構 bug、既有失敗、環境問題或測試需要同步調整。
9. 回報修改內容、行為保留方式、驗證結果、未驗證風險與必要的後續行動。

## 停止條件

滿足以下任一條件即可停止擴張範圍：

- 已完成使用者指定的重構目標，且相關驗證通過。
- 已完成最小可交付的重構，剩餘項目會擴大範圍或需要新的產品/架構決策。
- 發現必須改變行為、公開 API、資料格式或跨團隊契約才能繼續。
- 關鍵不確定性落在未提供的執行時資訊、資料庫內容、外部系統或使用者決策上。

## 預設忽略

除非與重構目標直接相關，否則不要優先修改：

- `node_modules`、`dist`、`build`、`.next`、`coverage`。
- lock file、編譯輸出與大型生成檔。
- 與目前入口無直接關聯的測試檔。
- 純樣式檔，除非重構目標涉及 UI class、樣式條件或元件邊界。
- 第三方函式庫原始碼。

## 重構類型

根據目標選擇策略。若需要更細的檢查清單，讀取 `references/refactoring-playbooks.md`。

- `rename`：重新命名變數、方法、類別、檔案或模組。先找引用點，確認外部契約與反射/字串引用。
- `extract`：抽取函式、類別、元件、hook、service 或 helper。先定義輸入、輸出、副作用與錯誤處理邊界。
- `split-module`：拆分過大的檔案或模組。先切出穩定責任邊界，不要同時重設架構。
- `deduplicate`：合併重複邏輯。先確認重複是否真的語意相同，而不是只有長得相似。
- `dependency-cleanup`：整理依賴方向、import、注入或耦合。先確認初始化順序與循環依賴風險。
- `dead-code-removal`：移除死碼。先搜尋動態引用、設定、模板、路由、反射與外部入口。
- `api-preserving-redesign`：內部重組但保留公開 API。先列出公開輸入、輸出、錯誤與相容性條件。

## 輸出指引

預設使用 `apply-in-place` 模式：直接修改專案中的目標檔案，例如 `.scss`、`.less`、`.vue`、`.ts`、`.js`、測試或相關 `.md` 文件。控制台只輸出摘要、檔案清單、驗證結果與風險，不要貼出完整大型檔案內容。

若使用者要求先產出候選重構、不要直接修改原檔、要先審核草案，或明確指定輸出目錄，使用 `artifact-dir` 模式：

- 預設輸出到 `refactor/<task-slug>/`，除非使用者指定其他目錄。
- 盡量保留原始相對路徑，讓檔案對應清楚，例如 `refactor/order-service/proposed/src/services/order.ts`。
- 補一份 `refactor/<task-slug>/README.md`，說明重構目標、檔案對應、如何套用、建議驗證命令與已知風險。
- 對說明文件、分析筆記、遷移步驟或大型輸出，優先寫成 `.md` 檔，不要全部貼在控制台。
- 控制台只回報 artifact 位置、重點摘要、驗證狀態與需要使用者決策的點。

回報應精簡但包含工程上必要的資訊：

- 改了哪些檔案與核心變更。
- 使用了 `apply-in-place` 或 `artifact-dir`，以及 artifact 位置若有。
- 如何保留既有行為。
- 執行了哪些驗證命令與結果。
- 哪些驗證沒有執行，以及原因。
- 若有，列出殘餘風險或需要使用者決策的點。
