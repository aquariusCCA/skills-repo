# Repair Study Notes

# 這個 skill 的任務

這個 skill 的任務不是把筆記排整齊，而是把原本難學、難懂、可能有錯的學習筆記，重構成真正能拿來學習、複習與回查的版本。

它處理的是**內容層修復**，不是只有版面整理。

---

# 何時使用這個 skill

當使用者要求修復、重構、改寫或校正某個學習筆記檔案，且符合以下任一情況時，使用本 skill：

- 原文表達難懂，讀者不容易抓到主線
- 同一篇內前後說法矛盾
- 例子、結論、定義彼此對不上
- 知識點順序混亂，導致理解成本高
- 內容摻雜錯誤知識、錯誤推論或錯誤術語
- 原文像口語逐字稿、想到哪寫到哪，難以作為學習筆記
- 原文雖然資訊很多，但不適合直接複習與回查
- 程式碼與文字解說互相衝突
- 原文重複過多，主線被噪音淹沒

典型請求：

```text
請用 repair-study-notes 重構 `XXX.md`
請幫我修復這篇筆記，原文有點亂
這篇筆記前後矛盾，請整理成可學習的版本
這份課程筆記像逐字稿，請重構
```

---

# 何時不要使用這個 skill

若原文內容本身清楚，只是缺少以下外層結構，則不要優先使用本 skill，而應改用一般整理型 skill，例如 `study-notes-markdown`：

* 缺標題
* 缺導讀
* 缺導航
* 缺 README
* 排版凌亂但內容本身正確且可讀
* 只需要補關鍵字、章節入口、回查索引

簡化判斷：

```text
repair-study-notes = 修內容
study-notes-markdown = 整入口、導航與倉庫結構
```

# 核心目標

整理完成後，筆記應盡量達成以下效果：

1. 讀者能快速知道這篇到底在講什麼
2. 讀者能分清楚定義、條件、推論、例子、例外
3. 讀者不會被原文中的矛盾與跳躍敘述卡住
4. 錯誤知識會被修正，或至少被明確標示
5. 內容順序更符合學習路徑
6. 複習時可以快速抓到核心
7. 後續擴充時不會因為原文混亂而越改越亂
8. 全文術語一致
9. 能順讀，也能回查

# 核心原則

## 1. 忠於原意，但不忠於原句

* 保留原本想表達的知識意圖
* 不必死守原句、原段順序、原本口語表達
* 允許把口語改成教學型表達
* 允許把一大段拆成多個清楚小節
* 允許把分散在不同位置的同類內容合併

## 2. 理解優先於形式保留

* 若原文形式妨礙理解，可改寫
* 若原文順序讓讀者難懂，可重排
* 若原文有重複、繞圈、插話，可刪減噪音
* 若原文像逐字稿，應轉成真正可讀的筆記

## 3. 先修內容，再補外層結構

處理優先順序固定如下：

1. 看懂原文在講什麼
2. 找出矛盾、錯誤、跳步、混淆點
3. 抽出知識骨架
4. 依骨架重寫內容
5. 最後才補導讀、關鍵字、回查入口、延伸閱讀

## 4. 錯誤不能直接沿用

若原文存在明顯錯誤，不得只是照抄整理，必須採取以下其中一種做法：

1. 直接修正為正確版本
2. 標示原文有誤，並給出修正版
3. 若無法確認正誤，明確標示「待查證」

## 5. 矛盾必須顯性處理

若同一篇筆記出現前後矛盾：

* 不可只保留其中一段而不說明
* 不可把互相衝突的內容並列卻不處理
* 應判斷哪一個版本較可靠
* 能判斷時，採用較可靠版本並統一全文
* 不能判斷時，明確標示衝突點與待確認處

## 6. 重構不是過度濃縮

* 不把完整教學內容縮成只有提綱
* 不把推理刪成只剩結論
* 不為了短而犧牲理解
* 重點是讓內容變清楚，不是變最短

## 7. 兼顧順讀與回查

重構後的內容應同時支援：

* 第一次閱讀時能一路讀懂
* 複習時能快速定位重點
* 忘記某個概念時能單點回查
* 想看例子時能快速找到例子
* 想比較概念差異時能快速找到比較段

---

# 判斷優先級

當原文出現衝突、混用或不確定時，依以下順序判斷哪個版本較可靠：

1. **明確定義是否一致**
2. **例子是否支持該說法**
3. **程式碼、流程或公式是否支持該說法**
4. **上下文前後是否一致**
5. **是否符合一般穩定知識與術語習慣**

若前 3 項無法支持任何一方，不要硬選，改列為「待查證」。

這個順序必須優先於主觀語感判斷。

# 內容問題類型與處理策略

## 1. 原文難懂

特徵：

* 句子太長
* 代詞過多，不知道在指什麼
* 一句話同時塞太多概念
* 抽象詞太多，沒有落地說明

處理方式：

* 改寫成短句
* 補出主語與條件
* 把一段拆成數個小步驟
* 先講結論，再講理由
* 必要時加入一句過渡解釋

## 2. 上下文矛盾

特徵：

* 前面說 A，後面又說非 A
* 同一術語在不同段落代表不同意思
* 前面定義與後面例子不一致
* 程式流程與口頭解說對不上

處理方式：

* 找出衝突點
* 依「判斷優先級」決定採信順序
* 統一術語、條件與結論
* 必要時保留修正說明

## 3. 錯誤知識

特徵：

* 事實性錯誤
* 概念性錯誤
* 推論邏輯錯誤
* 名詞定義錯誤
* 程式碼行為解讀錯誤

處理方式：

* 能確認時直接修正
* 補上正確說法
* 若可能誤導讀者，應明確標示
* 若不能確認，列入待查證

## 4. 敘述順序失衡

特徵：

* 先講例外，再講定義
* 先看程式碼，後面才補條件
* 一直跳來跳去
* 重要前置知識放在後面

處理方式：

優先重排成以下順序：

1. 這篇在解決什麼問題
2. 先講結論 / 核心理解
3. 核心定義 / 規則
4. 判斷條件 / 成立條件
5. 推理過程
6. 例子
7. 常見混淆點 / 易錯點
8. 一句話抓核心

## 5. 重複與噪音太多

特徵：

* 同一件事反覆說
* 有很多口語填充
* 舉例過多但沒有新增資訊
* 插入與主線無關的旁支內容

處理方式：

* 合併重複段
* 刪除無資訊增量的句子
* 保留最有代表性的例子
* 把旁支內容移到「補充」或「延伸」

# 強制工作流程

每次使用本 skill，必須依照以下流程處理，不可跳步：

## 第一步：判斷是否真的需要內容重構

符合以下任一情況，直接視為需要重構：

* 原文不是只缺結構，而是本身就難懂
* 讀完後無法清楚說出核心結論
* 同篇內出現明顯矛盾
* 有錯誤知識
* 原文像逐字稿，不適合直接作為筆記
* 例子和規則對不上
* 程式碼與文字解釋互相衝突

若只是缺標題、導讀、導航、README 或純排版問題，則不要用本 skill 當主要 skill。

## 第二步：抽出原文知識骨架

在正式改寫前，先辨識：

1. 這篇的主題是什麼
2. 作者真正想解決的問題是什麼
3. 核心定義有哪些
4. 條件與結論有哪些
5. 例子是在支持哪個規則
6. 哪些地方互相矛盾
7. 哪些地方可能有錯

若骨架尚未清楚，不要急著潤飾句子。

## 第三步：決定重構幅度

依內容狀況，只能選擇以下三種之一，並讓最終輸出與之相符：

### 輕度修復

適用於：

* 原文本身大致正確
* 幾乎沒有矛盾
* 只有少量句子難懂
* 只需少量澄清與整序

### 中度改寫

適用於：

* 主體其實正確，只是表達繞
* 內容完整，但順序不好
* 少數術語不一致
* 個別段落不清楚，但整體可救

### 大幅重寫

適用於：

* 原文像逐字稿
* 原文順序混亂
* 原文前後矛盾
* 原文存在錯知識
* 原文大量重複且難讀
* 原文難以直接作為教材或複習筆記

## 第四步：建立新結構

除非內容極短，否則優先使用以下骨架：

```md
# 標題

## 這篇在解決什麼問題

## 先講結論 / 核心理解

## 核心定義 / 規則

## 判斷條件或推理過程

## 例子說明

## 常見混淆點 / 易錯點

## 修正說明 / 待查證（如有）

## 一句話抓核心
```

若內容不需要某些段落，可以刪除；但不得省略前 2 項。

## 第五步：重寫內容

改寫時必須做到：

* 句子更短
* 指代更清楚
* 條件更明確
* 概念先後更合理
* 同一術語全篇一致
* 結論與例子能互相對上

## 第六步：最後補外層輔助結構

必要時再補：

* 本節導讀
* 關鍵字
* 建議回查情境
* 延伸閱讀
* 常見回查問題

# 強制最終輸出契約

只要使用本 skill，最終輸出至少必須包含以下四部分：

## A. 重構後正文

給出整理後、可直接閱讀與複習的版本。

## B. 原文主要問題摘要

用簡短條列說明原文主要問題，至少涵蓋以下可能項目中有出現者：

* 難懂
* 前後矛盾
* 錯誤知識
* 術語混用
* 順序失衡
* 重複噪音
* 程式碼與文字衝突

## C. 本次修正決策

至少交代：

* 採用的是輕度修復 / 中度改寫 / 大幅重寫
* 有無統一術語
* 有無刪除重複噪音
* 有無處理衝突
* 有無保留待查證處

## D. 待查證 / 不確定處（若有）

若存在無法確認的部分，必須明確列出。
若沒有，寫「本次未保留待查證項目」。

不得只輸出重構正文而省略 B、C、D。

# 衝突、修正與待查證格式

## 修正說明格式

```md
## 修正說明
- 原文將 A 說成 B，這裡已改為 C，因為後文例子 / 程式流程 / 定義不支持原說法。
```

## 待查證格式

```md
## 待查證
- 原文此處同時出現 A 與非 A 的說法，但現有內容不足以判斷哪個正確，暫列待查證。
```

## 衝突說明格式

```md
## 衝突處理
- 前段將 X 定義為 ...
- 後段又將 X 用作 ...
- 為維持全文一致，以下統一採用 ...
```

---

# 術語統一規則

* 同一概念全篇只能有一個主要術語
* 若原文存在多種說法，第一次出現時應說明對應關係
* 不可前面用 A、後面改用 B，卻不交代兩者是否同義
* 若術語本身可能錯，先修正，再統一

---

# 拆分規則

當原文很長，且同時混有多個子主題時，應拆成多篇，而不是硬塞在一篇重寫。

符合以下任一情況時，應優先考慮拆分：

* 同時涵蓋定義、流程、例子、延伸、實作
* 每個子主題都值得獨立回查
* 原文已經有明顯主題切換
* 不拆會讓單篇過長且難回查
* 一篇內同時處理超過 3 個明顯不同的核心問題

拆分後應盡量保留：

* 總入口頁
* 每篇單一主題
* 明確閱讀順序
* 哪篇看定義、哪篇看例子、哪篇看易錯點

若本次未拆分，也可在「本次修正決策」中註明不拆分原因。

---

# fallback 規則

## 1. 無法確認正誤時

不要硬改成肯定句，改列為「待查證」。

## 2. 無法完全判斷哪個衝突版本正確時

不要把兩者都寫成對的，應明確保留衝突資訊，並列入待查證。

## 3. 原文資訊不足時

可以整理成較穩定的教學骨架，但不得虛構原文沒有提供的關鍵條件、例子或結論。

## 4. 沒有模板檔可讀時

直接依本 skill 內建的「強制最終輸出契約」與「修正 / 待查證格式」輸出，不得因此省略修正說明。

---

# 成功標準

完成後，應盡量滿足：

1. 讀者可以用自己的話複述核心概念
2. 讀者能說出條件、結論與例子的關係
3. 全文術語一致
4. 前後沒有未處理的矛盾
5. 錯誤知識已修正或被明確標示
6. 內容既能順讀，也能回查
7. 比原文更適合學習，而不只是更好看
8. 最終輸出符合「強制最終輸出契約」

---

# 最終驗收清單

在交付前，至少檢查以下各項：

* 是否真的處理了內容問題，而不只是排版
* 是否明確說出原文主要問題
* 是否標出本次採用的重構幅度
* 是否統一全文術語
* 是否處理所有已發現的矛盾
* 是否把無法確認處列入待查證
* 是否讓例子、規則、結論互相對上
* 是否保留可順讀與可回查性
* 是否符合最終輸出契約

若其中任一項明顯不成立，不應視為完成。

---

# 禁止事項

* 不可把錯誤知識包裝得更流暢卻不修正
* 不可只調排版、不處理內容矛盾
* 不可將互相衝突的結論同時保留為正確
* 不可過度濃縮成只有考前速記
* 不可為了保留原味而犧牲可理解性
* 不可在無把握時假裝確定
* 不可省略原文問題摘要與修正決策
* 不可把待查證內容偽裝成已確認知識
