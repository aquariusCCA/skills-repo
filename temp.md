# 原始碼閱讀與專案理解 SKILL

## 1. 文件名稱與定位

### 1.1 Skill 名稱
原始碼閱讀與專案理解 SKILL  
Source Code Reading & Project Understanding Skill

### 1.2 文件定位
本 SKILL 用於閱讀專案原始碼、理解功能流程、建立模組認知，並輸出固定格式的結構化筆記。  
本 SKILL 的核心任務是「理解現況」，不是直接進行大規模重構。

### 1.3 核心用途
本 SKILL 適用於以下任務：

- 閱讀舊專案或陌生專案
- 解讀單一檔案的責任與邏輯
- 從入口追蹤某個功能的呼叫鏈與資料流
- 建立某個模組的整體理解
- 產出可累積、可複用的專案閱讀筆記
- 作為日後重構 SKILL、翻新 SKILL、技術遷移 SKILL 的前置理解層

### 1.4 非目標
本 SKILL 不負責以下任務：

- 不直接大規模重寫整個模組
- 不在資訊不足時假裝已理解整個專案
- 不一次掃描整個專案後直接下總結
- 不將猜測包裝成事實
- 不把「閱讀理解」與「重構設計」混在同一輪輸出中

---

## 2. 使用場景

### 2.1 適用情境
本 SKILL 適合以下場景：

- 接手舊專案，需要先看懂再修改
- 翻新前，需要建立功能流與模組認知
- 除錯前，需要先理解某條流程
- 新人進場，需要快速建立上下文
- 功能改版前，需要先確認原有商業邏輯與資料流
- 想把零碎讀碼結果整理成結構化筆記

### 2.2 適用專案類型
本 SKILL 不綁定單一技術棧，而是透過參數適配不同專案，例如：

- 舊式 Java Web / JSP / Servlet 專案
- Spring / Spring MVC / Spring Boot 專案
- Hibernate / MyBatis / MyBatis-Plus 專案
- Vue 2 / Vue 3 專案
- jQuery / 原生 JS / HTML / CSS 專案
- 前後端混合、分層不清、歷史包袱較重的舊系統

### 2.3 不適用情境
以下情境不適合直接使用本 SKILL 作為唯一方案：

- 使用者要求直接重構整個系統
- 沒有提供任何入口資訊，卻要求完整推論全專案架構
- 需要精準執行結果驗證，但沒有執行環境、日誌或測試資訊
- 主要需求是技術選型、架構設計或重構方案，而非理解現況

## 3. 核心工作原則

### 3.1 入口驅動原則
分析必須從指定入口出發，例如：

- 某個檔案
- 某個類別
- 某個方法
- 某個功能名稱
- 某個模組名稱

除非任務明確要求，否則不得直接嘗試掃描整個專案。

### 3.2 有限展開原則
只展開與目前任務直接相關的依賴與呼叫鏈。  
展開範圍應受 `max_depth` 控制，避免無限制擴散。

### 3.3 先理解再推論
優先整理目前程式中可直接確認的內容，再做有限度推測。  
推測必須清楚標註為推測，不能與已確認內容混淆。

### 3.4 不過度結論
若只閱讀局部檔案，則只能對局部下結論。  
不得因單檔內容直接推定整個模組或整個系統的完整設計。

### 3.5 每輪都要產出筆記
每輪分析都必須輸出固定格式的結構化筆記，而不是只給零散說明。

### 3.6 區分資訊狀態
每次分析都必須區分以下三類資訊：

- 已確認：可以直接從目前看到的程式內容得出
- 合理推測：根據命名、目錄、呼叫關係做出的有限推論
- 待驗證：目前證據不足，需進一步閱讀其他檔案才能確認

### 3.7 嚴守證據邊界
沒有讀到的檔案，不得假裝已閱讀。  
沒有看到的 SQL、API、設定、Mapper、畫面邏輯，不得補腦成完整事實。

## 4. 輸入參數設計

本 SKILL 支援參數化，以適配不同技術棧與不同分析任務。

### 4.1 必要參數
至少需要下列其中一組入口資訊：

- `analysis_mode`
- `entry_file`
- `target_feature`
- `target_module`

### 4.2 專案技術參數
可傳入以下技術背景，幫助分析時採用更合適的觀察角度：

- `project_type`
- `frontend_stack`
- `backend_stack`
- `database_type`

#### 範例
- `project_type: legacy-java-web`
- `frontend_stack: html, css, js, jquery`
- `backend_stack: java, springmvc, hibernate`
- `database_type: oracle`

或：

- `project_type: vue-springboot`
- `frontend_stack: vue3, typescript, vite`
- `backend_stack: springboot, mybatis-plus`
- `database_type: mysql`

### 4.3 分析範圍參數
- `entry_file`
- `entry_class`
- `entry_method`
- `target_feature`
- `target_module`
- `max_depth`

### 4.4 輸出控制參數
- `output_language`
- `note_style`

建議值：

- `output_language: zh-TW`
- `note_style: structured`

### 4.5 參數缺省規則
若部分參數未提供，應採取保守降級策略：

- 若未提供 `project_type`，可由檔案內容與命名推測技術類型，但要標註為推測
- 若未提供 `max_depth`，預設只追 1 層直接依賴
- 若未提供 `output_language`，預設輸出 `zh-TW`
- 若未提供 `note_style`，預設使用 `structured`
- 若只有 `entry_file`，則優先使用單檔解讀模式
- 若只有 `target_feature`，則優先使用功能追蹤模式
- 若只有 `target_module`，則優先使用模組總覽模式

## 5. 分析模式定義

本 SKILL 固定支援三種模式。

### 5.1 模式一：單檔解讀（file）

#### 目標
理解單一檔案的責任、角色、主要結構、依賴、上下游關係與核心邏輯。

#### 適用情境
- 想先看懂某個檔案在做什麼
- 想確認某個檔案屬於哪一層
- 想知道下一步應該追哪些檔案

#### 常見輸入
- `analysis_mode: file`
- `entry_file: ...`

#### 主要輸出
- 檔案定位
- 檔案職責
- 關鍵方法或區塊
- 依賴與互動
- 商業規則
- 風險與疑點
- 下一步閱讀建議

---

### 5.2 模式二：功能追蹤（feature）

#### 目標
從功能入口出發，追蹤呼叫鏈、資料流、驗證邏輯、查詢與更新流程。

#### 適用情境
- 想搞懂某個按鈕、畫面、API、報表、批次流程
- 想看某個功能如何串前後端
- 想知道資料從哪裡來、經過哪些轉換、最後寫到哪裡

#### 常見輸入
- `analysis_mode: feature`
- `target_feature: ...`
- `entry_file: ...`
- `entry_method: ...`
- `max_depth: 2`

#### 主要輸出
- 功能入口
- 流程總覽
- 呼叫鏈
- 涉及檔案
- 資料流
- 關鍵商業邏輯
- 風險點
- 尚待確認項目
- 下一步閱讀建議

---

### 5.3 模式三：模組總覽（module）

#### 目標
建立模組級的整體理解，整理主要檔案分工、模組責任與關鍵耦合。

#### 適用情境
- 已讀過數個檔案，想整理成模組視角
- 想為日後翻新建立模組地圖
- 想知道模組邊界與核心類別分工

#### 常見輸入
- `analysis_mode: module`
- `target_module: ...`

#### 主要輸出
- 模組責任
- 模組內主要檔案與分工
- 典型流程
- 關鍵依賴
- 耦合與風險
- 建議補讀檔案

---

## 6. 技術棧感知規則

本 SKILL 應依據輸入技術棧切換分析重點。

### 6.1 前端：jQuery / 原生 JS / HTML 專案
分析時優先注意：

- DOM 綁定位置
- 事件註冊方式
- 頁面初始化流程
- AJAX 呼叫點
- 全域變數使用情況
- 公用 util 與共用函式耦合
- HTML 片段拼接與模板生成邏輯
- 表單欄位與參數組裝流程

### 6.2 前端：Vue 專案
分析時優先注意：

- component 邊界
- props / emits
- composables / hooks
- store 狀態管理
- router 路由對應
- API 呼叫位置
- 畫面渲染與狀態同步
- 頁面與共用元件的責任分工

### 6.3 後端：Spring / Spring Boot
分析時優先注意：

- Controller → Service → Repository / Mapper 分層
- DI / 注入關係
- DTO / VO / Entity / BO 的流動
- 交易邏輯
- validation
- 設定檔與外部整合
- 例外處理與共用攔截邏輯

### 6.4 後端：舊式 Java Web / Hibernate
分析時優先注意：

- Servlet / Action / Controller 入口
- JSP 跳轉與 request / response 處理
- Session / RequestScope / PageScope 使用
- DAO / HQL / SQL / Stored Procedure 呼叫點
- util 類的跨層耦合
- 程式與畫面、SQL 混雜的區域
- XML 設定、舊式框架慣例與命名

### 6.5 資料庫層
分析時優先注意：

- SQL 寫在哪裡
- 查詢條件如何組裝
- insert / update / delete 落點
- transaction 邊界
- Stored Procedure / Package / Trigger / View 依賴
- Entity / DTO / result mapping 對應

## 7. 標準分析流程

每次分析都應遵循下列流程。

### 7.1 確認本輪目標
先明確界定本輪是：

- 單檔解讀
- 功能追蹤
- 模組總覽

### 7.2 確認入口
指出本輪從哪裡開始：

- 檔案
- 類別
- 方法
- 功能入口
- 模組入口

### 7.3 判定角色
先判斷目前檔案或節點屬於哪一層，例如：

- UI
- Component
- Controller
- Service
- DAO / Repository / Mapper
- Entity / DTO / VO
- Config
- Utility

### 7.4 抽取主要結構
整理以下內容：

- 類別 / 元件 / 函式
- 重要方法
- 關鍵欄位 / 狀態
- 對外暴露介面
- 初始化流程

### 7.5 抽取依賴與互動
找出：

- 它依賴誰
- 誰可能呼叫它
- 它往下呼叫了誰
- 是否依賴外部 API、DB、共用工具、設定檔

### 7.6 整理邏輯與資料流
釐清：

- 處理流程
- 參數來源
- 中間轉換
- 驗證邏輯
- 資料查詢與更新點
- 結果回傳與渲染方式

### 7.7 區分資訊狀態
將發現分為：

- 已確認
- 合理推測
- 待驗證

### 7.8 產出固定格式筆記
依照對應模式的模板輸出，不可自由散寫。

### 7.9 提出下一步建議
指出下一輪應優先閱讀哪些檔案，並說明原因。

---

## 8. 證據等級與推論規範

### 8.1 已確認
僅能包含下列類型資訊：

- 目前檔案中明確存在的方法、欄位、流程
- 明確 import、呼叫、注入、引用到的類別或函式
- 明確出現的 SQL、API、設定、參數組裝
- 程式內直接可見的條件判斷與商業規則

### 8.2 合理推測
只有在下列情況下才能推測：

- 根據命名慣例推測層級角色
- 根據目錄結構推測模組歸屬
- 根據呼叫關係推測上下游責任
- 根據框架慣例推測一般用途

所有推測都必須明確標示為「推測」，不得混入已確認結論。

### 8.3 待驗證
若出現以下情況，應歸入待驗證：

- 只看得到呼叫點，看不到實作
- 只看得到 SQL 名稱，看不到 SQL 本體
- 只看得到畫面事件，看不到 API 或後端
- 只看得到 Service 呼叫，看不到 DAO / Mapper
- 只看得到欄位名稱，看不到來源定義
- 多種流程可能性並存

### 8.4 禁止事項
- 不可把命名推測當成事實
- 不可因檔名像 Service 就直接斷言完整商業責任
- 不可因一個 Controller 的存在就推定整個功能流程
- 不可在未看到 SQL 時斷言查詢條件完整內容
- 不可在未讀到其他模組時斷言模組邊界
- 不可把猜測寫成肯定句

---

## 9. 單檔解讀輸出模板

當 `analysis_mode = file` 時，輸出必須遵循以下格式。

### 9.1 基本資訊
- 檔案路徑
- 檔名
- 技術類型
- 所屬模組
- 推定層級

### 9.2 檔案職責
用 2 至 5 句話說明：

- 這個檔案在做什麼
- 它在整體系統中的角色
- 它對外承擔什麼責任

### 9.3 主要結構
整理：

- 類別 / 元件 / 函式
- 重要方法
- 關鍵欄位 / 狀態
- 初始化流程
- 對外暴露能力

### 9.4 核心邏輯
說明：

- 主要處理流程
- 關鍵條件判斷
- 參數組裝
- 驗證邏輯
- 結果輸出或回傳方式

### 9.5 依賴與互動
列出：

- 上游：誰可能呼叫它
- 下游：它呼叫了誰
- 外部依賴：DB、API、共用 util、設定等

### 9.6 商業規則
整理具有業務意義的規則，例如：

- 欄位判斷
- 狀態轉換
- 身分條件
- 權限條件
- 金額、日期、流程分支規則

### 9.7 風險與疑點
指出：

- 命名不清
- 多責任混雜
- 方法過長
- SQL / UI / 邏輯混在一起
- 框架耦合
- 全域狀態過多
- 目前無法確認的區域

### 9.8 已確認 / 合理推測 / 待驗證
必須分開列出。

### 9.9 下一步建議
列出下一輪最值得閱讀的檔案，並標註原因。

---

## 10. 功能追蹤輸出模板

當 `analysis_mode = feature` 時，輸出必須遵循以下格式。

### 10.1 功能名稱
簡潔描述本輪要追的功能。

### 10.2 功能入口
標註：

- 頁面 / 按鈕 / API / 排程 / 批次
- 起始檔案
- 起始方法
- 起始事件

### 10.3 流程總覽
用分段或條列方式描述整體流程。

### 10.4 涉及檔案清單
列出本輪涉及的核心檔案，並簡述其責任。

### 10.5 呼叫鏈
用鏈路方式表示，例如：

頁面事件  
→ 前端函式  
→ API 呼叫  
→ Controller  
→ Service  
→ DAO / Mapper  
→ SQL / Package / DB  
→ 回傳整理  
→ 前端渲染

### 10.6 資料流
整理：

- 輸入資料從哪裡來
- 中間如何轉換
- 驗證在哪裡做
- 查詢或更新資料落在哪裡
- 輸出結果如何回到畫面或呼叫端

### 10.7 關鍵商業邏輯
整理整條功能鏈中真正重要的規則。

### 10.8 風險點
指出：

- 流程過長
- 驗證分散
- SQL 分散
- 多處共用狀態
- 邏輯跨層
- 舊框架耦合
- 交易邊界不清
- 命名與實際行為不一致

### 10.9 已確認 / 尚未確認
明確分開列出。

### 10.10 下一步建議
指出下一輪優先要補讀的檔案、SQL、Mapper、設定或畫面事件。

---

## 11. 模組總覽輸出模板

當 `analysis_mode = module` 時，輸出必須遵循以下格式。

### 11.1 模組名稱
### 11.2 模組目標與責任
### 11.3 模組內主要檔案與分工
### 11.4 典型流程
### 11.5 關鍵依賴
### 11.6 與其他模組的互動
### 11.7 耦合與風險
### 11.8 建議補讀檔案
### 11.9 後續理解建議

模組總覽必須避免空泛描述，應根據目前已閱讀內容保守整理，不得假裝已掌握整個模組的所有實作。

---

## 12. 下一步閱讀建議規則

每輪分析結尾都要提出下一步建議，但必須遵守以下規則。

### 12.1 優先追的內容
優先追以下對象：

1. 當前檔案直接呼叫的核心檔案
2. 呼叫當前檔案的明確入口
3. 關鍵 SQL / Mapper / DAO / Repository
4. DTO / VO / Entity 定義
5. 與功能直接相關的設定檔或共用 util
6. 與流程分支直接相關的畫面事件或後端方法

### 12.2 優先順序規則
若需要排序，原則如下：

1. 功能入口
2. 核心商業邏輯
3. 資料落點
4. 參數轉換節點
5. 共用工具
6. 次要輔助檔案

### 12.3 禁止無限制擴散
不得因為看到 import 很多檔案，就把所有檔案都列為下一步。  
只保留與本輪目標直接相關的內容。

---

## 13. 資訊不足與中斷處理

### 13.1 入口不完整
若只有檔名，沒有方法名或功能名，則先做保守單檔解讀，不得強行拼出完整流程。

### 13.2 依賴鏈中斷
若只看到呼叫點，看不到被呼叫實作，則必須：

- 明確指出中斷點
- 說明目前無法確認的內容
- 將該檔案列入下一步建議

### 13.3 命名不清
若命名與職責不一致，或命名過於籠統，應列為風險與疑點，不可硬判。

### 13.4 多條可能流程並存
若一個功能存在多種分支可能性，應分開列出，不能合併成單一路徑。

### 13.5 檔案內容過長
若單檔過大，應優先抓：

- 入口方法
- 主要流程
- 關鍵條件分支
- 對外互動點

其餘內容列為次要區塊，不需假裝已完整消化所有細節。

## 17. 限制與邊界

### 17.1 本 Skill 的限制

本 SKILL 有以下限制：

* 無法替代真實執行結果
* 無法憑少量檔案保證完整理解整個系統
* 無法替代測試、日誌、執行時觀察
* 若入口資訊不足，只能做局部理解與保守推測

若可取得以下資訊，分析品質會更高：

* 專案目錄樹
* 入口檔案
* 功能名稱或操作步驟
* API 文件
* 資料表結構
* SQL / Package / Procedure
* 已知錯誤訊息或業務背景
