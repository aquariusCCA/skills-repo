# Study Notes Markdown

當使用者希望保留教材型、教學導向、較詳細的正文內容，但又希望筆記在後續更容易理解、瀏覽、搜尋、跳讀、回查與複習時，使用這個 skill。

這個 skill 的核心不是把內容濃縮成摘要，而是讓原本的筆記「更好用」。

這裡的「更好用」不是單一目標，而是同時提升以下四件事：

1. 更容易理解
2. 更容易閱讀
3. 更容易查找
4. 更容易複習

若四者衝突，優先順序是：

1. 理解
2. 表述精度
3. 回查效率
4. 結構一致性
5. 版面整齊

若模板化會傷害理解或精度，應允許偏離模板。

---

# 適用情境

以下情境適合套用本 skill：

- 課程筆記、教材筆記、技術學習筆記
- 想保留完整教學型正文，而不是做極短摘要
- 單篇筆記過長、缺乏導航、閱讀中容易迷路
- 想把零散筆記整理成可持續擴充的 Markdown 倉庫
- 想讓筆記同時支援順讀、跳讀、搜尋、回查與複習
- 想把原本像 Notion 長頁、隨手記、教材摘錄的內容，整理成章節化知識庫

# 不適用情境

以下情境不應直接套用，或需調整後再套用：

- 使用者只要極短摘要、考前一頁速記、口訣表
- 使用者要的是文章改寫、潤稿、翻譯，而不是筆記整理
- 使用者要的是知識卡片系統，而不是教材型長文筆記
- 原文本身明顯錯亂、錯誤很多、缺乏基本邏輯，需要先重構內容本身

若使用者要的是「考前速記版」，應在保留本 skill 主體的前提下，另外產出一份獨立的複習稿，而不是把正文直接壓縮掉。

# 輸入契約

當使用者要求整理 Markdown 學習筆記，且目標是提升理解、閱讀、查找、回查或複習效率，而不是單純濃縮成摘要時，應套用本 skill。

可接受的輸入形式包括：

- 明確提到 `study-notes-markdown`
- 雖未提到 skill 名稱，但需求描述符合本 skill 目的
- 指定整理單篇筆記、章節筆記或整份筆記倉庫

使用者不需要在 Markdown 檔案中加入特殊語法，也不需要在終端執行指令。

# 核心目標

不要刪掉核心教學內容。  
不要把詳細說明改成只有提綱。  
不要為了整齊而犧牲理解。  
不要為了複習方便而只留下結論。  
不要讓筆記只剩好看，卻不好用。  
不要讓搜尋只能靠運氣命中。  
不要讓複習還得重讀整篇才能找到重點。

重點是替既有內容補上穩定的外層結構，讓筆記同時支援：

- 第一次學習時可順讀
- 複習時可快速定位
- 搜尋時容易命中
- 跳讀時不容易迷路
- 回查時能快速判斷是否找對頁
- 之後擴充時維持一致性
- 需要自測時能快速提取問題
- 考前需要快速掃描時有固定入口

# 這個 skill 要解決的問題

常見原始筆記問題包括：

- 單篇過長，讀到中途容易失去定位
- 缺乏章節導航，不知道先看哪篇
- 回查時只能全文搜尋，效率低
- 筆記保留了很多內容，但不容易快速抓到入口
- 內容本身不一定差，但結構不足，導致難讀、難找、難複習
- 筆記裡有重要定義、公式、前提、例外，但沒有穩定標記
- 搜尋詞不一致，例如同一概念用多個叫法，導致查找不穩
- 複習時不知道應該先看哪個區塊，也不知道哪些是高頻回查點

這個 skill 的工作重點是補結構、補入口、補導航、補回查線索、補速查支架、補搜尋命中線索。  
不是把所有內容都重寫成精簡版。

# 成功標準

整理完成後，筆記應盡量達成以下效果：

1. 使用者一打開就知道這篇在講什麼
2. 使用者知道這篇適合怎麼讀
3. 使用者知道這篇和前後知識點的關係
4. 使用者之後想回查時，能在短時間內找到答案
5. 使用者複習時，不必重讀整篇才能抓到重點位置
6. 使用者在倉庫中不容易迷路
7. 原本重要的定義、前提、限制、例外與推導沒有因整理而消失
8. 模板化後，內容仍保有原本應有的表述精度
9. 同義詞、別名、英文術語等能提升搜尋命中率
10. 常見查詢意圖能透過速查區、索引或 FAQ 快速命中
11. 高價值內容如定義、流程、對比、易錯點能被快速掃到
12. 筆記可同時服務順讀者與回查者，而不是只服務其中一種人

# 核心原則

## 1. 保留教材型正文

- 除非使用者明確要求重寫，否則保留原本教學型正文
- 不用摘要取代正文
- 不把段落改成只剩大綱式速記
- 允許做的是加上外層結構、導航、標示與少量輔助

## 2. 先補結構，再碰內容

整理順序優先是：

1. 倉庫入口與章節導航
2. 先判斷是否需要拆分大筆記
3. 單篇筆記開頭格式統一
4. 補高價值的速查區與關鍵字系統
5. 補必要的內部連結與索引
6. 視需要補複習輔助與自測入口
7. 最後才是小幅內容微調

## 3. 小改穩改，不過度濃縮

- 優先做可重複套用的結構化修改
- 不做大量語意重寫
- 不追求把每篇都整理成考前速記
- 不追求每段都加總結，避免破壞原文節奏
- 不以壓縮字數作為整理成功標準

## 4. 理解優先於整齊

若形式上的整齊會讓內容更難懂，應保留原本較易理解的寫法。  
整理的目的是讓筆記更容易吸收，而不是只讓版面變漂亮。

## 5. 模板是輔助，不是硬格式

- 模板可以套用，但不能機械式硬套
- 可依主題、內容密度、長度、用途微調
- 若模板會破壞內容的閱讀流暢度，可調整或省略局部欄位
- 不應為了欄位完整，而強行改壞原文節奏

## 6. 結構一致性要服務於使用，而不是反過來支配內容

- 一致性很重要，但應服務於閱讀、查找與回查
- 如果同一種模板套在所有筆記上反而造成干擾，應允許差異化處理

## 7. 查找能力不是附屬功能，而是正文的第二入口

- 好的筆記不只要能讀，還要能找
- 關鍵字、別名、英文術語、常見問法、索引，都是查找入口的一部分
- 搜尋命中率不應完全依賴使用者剛好輸入正確詞彙

## 8. 複習輔助應幫助提取，不是只幫助重看

- 複習不是單純把文章再看一次
- 筆記應提供固定的速查入口、回查情境、自測提示、易錯點提示
- 複習輔助應盡量不破壞正文節奏，但要讓提取更容易

# 工作流程

1. 檢查整體倉庫結構與章節分布
2. 判斷目前是大筆記還是小筆記
3. 若為大筆記，先拆分，再整理拆分後的小筆記
4. 若為小筆記，直接整理
5. 先補 README / 導航 / 開頭格式，並檢查根目錄 `README.md` 的目前進度與章節入口是否需要同步
6. 再補高價值內部連結、索引與關鍵字系統
7. 視需要補速查區、複習輔助、自測入口
8. 最後再做少量內容微調
9. 最後檢查是否因整理而損失表述精度
10. 最後檢查是否真的提升理解、閱讀、查找、複習四個面向

## 執行上的硬規則

- 若已判定為大筆記，不要只補文首導讀就結束；應先完成拆分，再整理拆分後的檔案
- 若同時符合大筆記條件與小筆記條件，優先按大筆記處理
- 若最終決定不拆，必須明確說明不拆的理由，例如篇幅其實可控、主題高度集中、拆分反而破壞理解
- 若拆分後會產生一組子筆記，應建立對應入口頁 `README.md`，而不是只散落多個檔案
- 若判定應補速查區、關鍵字或回查入口，不得只補標題而略過這些高價值欄位
- 若同義詞、英文術語或常見問法明顯存在，不應省略關鍵字系統

# 大筆記與小筆記的判斷

## 視為大筆記的情況

符合以下任一情況時，應視為大筆記並先拆分：

- 單篇過長，閱讀與搜尋成本高
- 同時涵蓋多個明顯子主題
- 已有穩定的標題階層，例如 `#`、`##`、`###`
- 使用者在單篇中不容易快速定位內容
- 單篇已同時承擔總覽、定義、比較、實作、例子等多種任務

## 建議的判斷門檻

符合以下任一門檻時，預設應拆分，除非有明確理由證明不拆更好：

- 單篇超過 `1500` 行
- 單篇有 `5` 個以上一級主題，或 `10` 個以上二級主題
- 同一篇同時包含「概念講解」與「多個獨立案例 / 題解 / 實作主題」
- 單篇已經接近一本小冊，讀者難以靠搜尋與標題快速定位

## 視為小筆記的情況

符合以下情況時，可直接整理：

- 單篇主題明確且範圍集中
- 長度可控
- 適合獨立閱讀與獨立回查
- 不拆分也不影響理解與維護

## 小筆記的補充判準

- 通常可在一次閱讀中消化，不需要頻繁上下跳讀
- 即使補上導讀、關鍵字與少量導航後，也已足夠支援回查
- 拆分後不會明顯增加理解收益，反而可能造成內容過碎

## 拆分時的實作規則

- 先沿用原本穩定的標題階層拆分，不要為拆而拆
- 優先拆成「主題資料夾 + README + 多篇小筆記」的形式
- 拆分資料夾預設使用原筆記檔名去掉 `.md` 後的名稱，例如 `2 链表.md` 拆成 `2 链表/`
- 拆分後的入口頁應放在該資料夾的 `README.md`
- 若原筆記已經有章節層級的 `images/` 目錄，拆分後的小筆記預設沿用原圖片，不複製資產
- 每篇拆出的子筆記都應能獨立閱讀，並保有必要的上下文
- 拆分後的檔名應具可回查性，例如 `01 概述.md`、`02 单向链表.md`、`06 LeetCode 206 反转链表.md`
- 拆分後應補上前後閱讀順序與返回入口連結，避免讀者從單篇進入後迷路
- 若原文中的同一主題有明顯的概念篇、例題篇、實作篇，優先分開而不是混在同一檔

## 拆分資料夾命名的例外

- 若原檔名過於泛化，例如 `README.md`、`笔记.md`、`总结.md`，可改用更具主題性的資料夾名
- 若同名資料夾已存在，先判斷是否本來就是這篇筆記的拆分目錄；若不是，應改用更清楚且不衝突的名稱
- 若原檔名已帶有明確序號，拆分後應保留該序號，避免破壞原本閱讀順序

# 倉庫層級設計

整份筆記倉庫至少應有三層入口：

1. 根目錄入口
2. 章節入口
3. 單篇入口

若缺任一層，閱讀與回查效率通常會明顯下降。

## 根目錄入口負責

- 告訴讀者這份筆記在學什麼
- 告訴讀者目前整理到哪裡
- 告訴讀者該從哪裡開始
- 告訴讀者有哪些高頻回查入口

## 章節入口負責

- 告訴讀者這章在學什麼
- 告訴讀者章內閱讀順序
- 告訴讀者哪些頁面適合速查
- 告訴讀者本章與前後章關係

## 單篇入口負責

- 告訴讀者這篇處理什麼問題
- 告訴讀者這篇適合何時讀
- 告訴讀者能查到哪些資訊
- 告訴讀者高價值段落在哪裡

---

# 根目錄 README

讓根目錄 `README.md` 成為整份筆記的入口頁，而不是只放標題。

根目錄 README 應優先包含：

1. 倉庫用途簡介
2. 目前學習進度
3. 章節導航
4. 如何使用這份筆記
5. 需要快速回查的入口
6. 常見查詢入口
7. 參考資料

## 根目錄 README 建議內容

- 這份筆記在學什麼
- 目前已整理到哪裡
- 各章節連結
- 建議閱讀順序
- 適合初學順讀的章節
- 適合複習跳讀的章節
- 高頻查找主題入口
- FAQ / 索引 / 術語表入口
- 參考資料放在導航之後，而不是之前

---

# 章節 README

每個主要章節資料夾都應有自己的 `README.md`。

章節 README 應包含：

- 本章主要內容
- 本章和整體學習地圖的關係
- 建議閱讀順序
- 本章所有小節連結
- 哪些主題適合快速回查
- 哪些主題是基礎概念
- 哪些主題最容易混淆
- 返回根目錄 README 的連結

## 章節 README 的用途

章節 README 不只是列表頁，它應該回答：

- 這章在學什麼
- 先讀哪篇比較不會卡
- 哪幾篇是概念基礎
- 哪幾篇是常回查頁
- 哪幾篇有公式、流程、對比、例題
- 這章讀完後通常接哪一章

---

# README 同步規則

整理單篇筆記時，應檢查該筆記所在章節的 `README.md`：

- 若章節 `README.md` 不存在，且單篇筆記使用了 `所属章节：[章節名](./README.md)`，應建立章節 `README.md`
- 若章節 `README.md` 已存在，但未包含本篇筆記，應同步補上本篇連結
- 若整理後改動了標題、檔名、閱讀順序、延伸閱讀或回查入口，應同步更新章節 `README.md`
- 若章節 `README.md` 在 Git 狀態中顯示為刪除，應明確說明此狀態；若整理後會造成壞鏈，應重建或請使用者確認

整理單篇筆記時，也應檢查根目錄 `README.md` 的目前進度、章節導航與快速入口是否仍準確。  
根目錄 `README.md` 不需要每次整理單篇都機械式改動，但只要符合以下情況，就必須同步：

- 新增、刪除、移動章節資料夾
- 新增章節 `README.md`
- 章節導航或整體學習路徑發生變化
- 根目錄 README 的章節入口缺失或已失效
- 根目錄 README 的目前進度與已整理小節不一致
- 根目錄 README 未列出已存在且應作為入口的章節或小節
- 單篇整理後新增、移動、改名或補齊章節入口，導致總目錄需要更新才能避免讀者迷路
- 單篇整理後新增了高頻速查入口、FAQ、索引頁，導致總目錄應補入口

若檢查後決定不更新根目錄 `README.md`，應能說明理由，例如總目錄已包含該章入口、目前進度仍正確，且本次整理沒有改變章節導航或整體學習路徑。

# 單篇筆記開頭格式

每篇筆記應優先補齊以下開頭結構：

1. 標題
2. 所屬章節
3. 本節導讀
4. 關鍵字
5. 建議回查情境
6. 30 秒複習入口
7. 速查區
8. 正文教學內容
9. 延伸閱讀 / 前後連結

## 單篇筆記開頭格式說明

### 1. 標題

- 標題應清楚反映主題，不用過度文學化
- 優先讓未來的自己能看懂，而不是追求短
- 若為題解或案例，標題應含題號、名稱或問題類型

### 2. 所屬章節

- 提供回到章節 README 的連結
- 若屬於多章交界主題，可附上主要章節與次要相關章節

### 3. 本節導讀

- 用 2 至 5 句說明這篇在處理什麼
- 說明適合在什麼背景下閱讀
- 可提示與前面知識的關聯

### 4. 關鍵字

關鍵字應盡量結構化，不應只寫 3 個任意詞。建議至少包含：

- 主關鍵字
- 同義詞 / 別名
- 英文術語
- 常見搜尋詞
- 容易混淆的相近詞

### 5. 建議回查情境

這個欄位要回答：

- 什麼時候我會回來查這篇
- 我通常是因為哪個問題回來
- 哪些任務、題型或工作情境會用到這篇

### 6. 30 秒複習入口

這不是摘要全文，而是給未來自己快速重啟記憶的入口。建議包含：

- 這篇最核心的 1 至 3 件事
- 最容易搞混的點
- 最常被拿來考或使用的點

### 7. 速查區

速查區應優先收納以下高頻回查項：

- 核心定義
- 關鍵公式 / 規則 / 判準
- 前提條件
- 例外情況
- 常見錯誤
- 一句話對比
- 常見問答

### 8. 正文教學內容

- 保留原本教學型正文
- 不用速查區取代正文
- 若正文已存在良好標題結構，優先沿用

### 9. 延伸閱讀 / 前後連結

- 提供上一篇、下一篇、前置知識、延伸主題
- 讓順讀和跳讀都不容易迷路

# 查找系統設計

好筆記不只要能讀，還要能找。  
查找系統至少應包含以下四種入口中的兩種；若為大型倉庫，建議至少三種：

1. 關鍵字欄位
2. README 導航
3. 索引 / FAQ / 術語表
4. 內部連結與前後篇連結

## 關鍵字規則

- 關鍵字必須服務搜尋，而不是只服務美觀
- 同一概念若有中英文、縮寫、別名，應盡量補齊
- 若常見搜尋詞與正式術語不同，應同時保留
- 若存在高混淆詞，應在關鍵字或易混淆區中明確標示

### 建議格式

- 主題：xxx
- 別名：xxx、xxx
- 英文：xxx
- 常見搜尋：xxx、xxx
- 易混淆：xxx vs xxx

## FAQ / 索引 / 術語表的使用時機

符合以下情況時，應考慮建立：

- 同一術語在多篇出現
- 使用者常用問句而不是術語來查找
- 同義詞很多，搜尋命中不穩
- 章節內容多，僅靠 README 很難快速定位

## 常見查詢詞前置原則

若未來使用者很可能會用以下方式搜尋，應在筆記中前置這些詞：

- 問句式，例如「什麼是...」「差別是什麼」「什麼時候用...」
- 縮寫式，例如 API、MVC、ACID、CAP
- 中英夾雜式，例如 索引 index、事務 transaction
- 題型式，例如 判斷條件、流程、公式、例外、比較

---

# 複習系統設計

本 skill 不是把所有筆記變成考前小抄，但應提供穩定的複習支架。

## 最低限度的複習支架

若整理單篇筆記，至少應補：

1. 建議回查情境
2. 30 秒複習入口
3. 速查區

若整理章節或整份倉庫，建議再補：

4. 章節高頻回查頁列表
5. 易混淆主題列表
6. 章節複習路線
7. FAQ / 自測入口

## 複習支架的原則

- 幫助提取，不取代正文
- 幫助定位，不製造第二份互相矛盾的內容
- 盡量固定位置，降低每次重新適應成本
- 複習欄位應偏短，但不能失真

## 可加入的複習元件

視主題需要，可加入以下其中幾項：

- 30 秒複習入口
- 一頁速查表
- 本篇最常錯的 3 點
- 自測問題 3 至 5 題
- 對比表
- 判斷流程摘要
- 常見陷阱
- 公式與適用條件對照

## 自測問題的規則

- 自測問題應偏向提取，不要只是讓人再看一次答案
- 問題要對應正文內容，而不是另外發明知識點
- 若該主題重理解，可問「為什麼」「差別在哪」「什麼情境用」
- 若該主題重規則，可問「條件是什麼」「何時例外」「如何判斷」

---

# 內容標記規則

當正文內容中存在高價值資訊時，應考慮加上穩定標記。  
標記不是裝飾，而是加速掃描與回查。

## 建議標記類型

- `定义`：核心定義、正式描述
- `重点`：最重要的觀念、規則、判準
- `前提`：成立條件、適用範圍
- `例外`：不符合一般規則的情況
- `对比`：相近概念比較
- `流程`：步驟、判斷順序
- `易错`：常見誤解、陷阱
- `题解`：題目解法、思路、關鍵轉折
- `实务`：實作場景、工作情境、應用提醒

## 標記使用原則

- 僅標記高價值內容，不要整頁都上標記
- 標記應穩定，可跨不同主題複用
- 標記名稱盡量固定，不要同義詞混用
- 標記後的內容仍應保有原文脈絡，不要切碎到難讀

---

# 命名與連結規則

## 檔名規則

- 檔名應能反映主題與用途
- 優先考慮未來回查時是否容易辨識
- 若存在閱讀順序，檔名前可加序號
- 若為題解，檔名可包含題號與題名

## 標題規則

- 標題應盡量與檔名一致或高度對應
- 避免過度抽象的標題，例如 `補充`、`其他`、`想法`
- 若主題有正式名詞，應盡量使用正式名詞

## 連結規則

- 每篇筆記至少應能回到章節 README
- 若有前置知識或後續篇章，應優先加上前後連結
- 若文中提到另一篇已存在且高度相關的筆記，應考慮補內部連結
- 若整理後更名、移動、拆分，應同步檢查壞鏈

---

# 輸出契約

當執行本 skill 時，輸出不應只交付修改後內容，還應說明以下資訊：

1. 本次是整理單篇、章節，還是整份倉庫
2. 是否判定為大筆記，是否拆分
3. 若不拆，理由是什麼
4. 本次補了哪些入口，例如 README、關鍵字、速查區、索引、自測
5. 是否同步更新根目錄或章節 README
6. 是否發現命名、壞鏈、缺入口、搜尋詞不一致等問題

若使用者只要結果，不一定要長篇解釋，但內部應完成以上檢查。

---

# 反模式

以下做法違反本 skill 的目標，應避免：

- 把長文全部壓成提綱，導致失去教學性
- 只做漂亮排版，不補導航與入口
- 只補導讀，不補關鍵字與速查區
- 只有章節列表，沒有閱讀順序與回查入口
- 關鍵字欄位只寫幾個隨機詞，無法支援搜尋
- 為了模板完整而硬塞欄位，破壞原文節奏
- 把複習支架做成第二份彼此矛盾的內容
- 大筆記應拆不拆，導致單篇越整理越難用
- 內容明明存在中英別名與常見問法，卻完全沒補搜尋入口
- 每段都加標記或總結，導致版面密度過高、閱讀更累

# 品質檢查清單

完成整理後，至少檢查以下問題：

## 理解

- 這篇是否仍能順讀？
- 核心定義、推導、前提、例外是否還在？
- 模板有沒有壓壞原本教學節奏？

## 閱讀

- 一打開是否知道這篇在講什麼？
- 是否知道先看哪裡？
- 從任意一篇進入，是否容易迷路？

## 查找

- 是否有穩定的關鍵字系統？
- 是否補了中英術語、別名、常見搜尋詞？
- README、索引、FAQ、內部連結是否足夠？

## 複習

- 是否有建議回查情境？
- 是否有 30 秒複習入口？
- 是否有速查區？
- 是否能不重讀全文就找到高頻回查點？

## 結構

- 是否同步更新相關 README？
- 是否檢查過壞鏈與命名一致性？
- 若拆分了，是否有入口頁與返回路徑？

# 最終準則

這個 skill 的終極目標，不是讓筆記變得更像模板。  
而是讓筆記在保留知識密度與教學價值的前提下，變成真正可理解、可閱讀、可查找、可複習、可擴充的長期資產。
